CH 01

Introduction to Computer Networks

What is the Internet?

The Internet can be understood from two angles. From a "nuts and bolts" perspective, it is billions of connected devices (called hosts or end systems) linked together by communication links (fiber, copper, radio, satellite) and packet switches (routers and link-layer switches). From a "service" perspective, it is an infrastructure that provides services to applications — like the Web, email, streaming, and gaming — through a programming interface (the socket API).

Host / End SystemAny device at the network edge — PCs, phones, servers, IoT devices. They run network applications.
Communication LinkPhysical medium connecting devices. Transmission rate = bandwidth (bits per second). Types: fiber, copper (twisted pair, coax), radio, satellite.
Packet SwitchForwards packets (chunks of data) toward destination. Routers operate at network layer; link-layer switches at link layer.
Internet StandardsDefined in RFCs (Request for Comments) by the IETF (Internet Engineering Task Force). Anyone can propose an RFC.

What is a Protocol?

A protocol defines the format, order of messages exchanged between network entities, and the actions taken on transmission or receipt of a message or other event. Everything in the Internet — from a browser requesting a page (HTTP) to two computers establishing a connection (TCP SYN/SYNACK) — is governed by protocols.

Think of it like a human conversation: both parties must agree on the language, when to speak, and what responses are expected. Network protocols are just the machine equivalent.

Network Core: Packet Switching

The network core is the mesh of interconnected routers at the heart of the Internet. Packet switching is how data moves through it.

Store-and-Forward Transmission

Routers use store-and-forward: the entire packet must arrive at a router before any of it can be forwarded to the next link. This adds a transmission delay of L/R at every hop.

End-to-end delay (store-and-forward, N links, zero propagation delay): delay = N × (L/R) Where: L = packet length (bits), R = link transmission rate (bps) Intuition: each of the N links must fully receive the packet before forwarding

Queuing and Packet Loss

Each router has output buffers (queues) for each outgoing link. When packets arrive faster than the link can transmit them, they queue up. If the buffer is full, arriving packets are dropped (lost). This is a fundamental property of packet networks — unlike circuit switching, resources are not reserved.

Two Key Router Functions

Forwarding (Data Plane)
  • Move packet from router's input port to correct output port
  • Uses local forwarding table (header value → output link)
  • Hardware, nanosecond timescale
Routing (Control Plane)
  • Determine the route packets take from source to destination
  • Routing algorithms compute forwarding tables
  • Software, millisecond timescale

Network Core: Circuit Switching

In circuit switching, resources along the entire path are reserved for the duration of the call before communication begins. Traditional telephone networks (PSTN) use this model.

FDM — Frequency Division Multiplexing
  • Each call gets a dedicated frequency band for the whole link
  • All calls transmit simultaneously, each on its own frequency
  • Bandwidth wasted if a call is silent
TDM — Time Division Multiplexing
  • Link capacity divided into time slots (frames)
  • Each call gets a specific slot in every frame
  • Slot is idle if call has nothing to send (wasted)
★ Exam: Packet vs Circuit Switching [Fall 2024 Q1b]
  • Why packet switching wins for bursty data: Resources shared among all users. Example — 1 Mbps link, users need 100 kbps when active, active only 10% of time. Circuit switching: 10 users max. Packet switching: 35 users work fine because probability more than 10 active at once ≈ 0.0004.
  • Why circuit switching still matters: Guaranteed bandwidth, predictable delay. Better for real-time applications like old telephony (though VoIP now runs over packet networks).

Internet Structure: Network of Networks

End systems connect to the Internet through access ISPs. But access ISPs must be interconnected so any two hosts can communicate. The resulting structure has evolved into a tiered hierarchy:

Access ISPs — residential, campus, corporate ISPs. Connect end users to Internet.

Regional ISPs — aggregate multiple access ISPs in a geographic area. Connect access ISPs to global tier.

Tier-1 ISPs — national and international scope (e.g., Level 3, AT&T, NTT, Sprint). Connected to each other at IXPs (Internet Exchange Points) and via peering links. At the center of the Internet.

Content Provider Networks — Google, Microsoft, Akamai run their own global networks to bring content close to users, bypassing tier-1 ISPs where possible.

Delay, Loss, and Throughput

The Four Sources of Packet Delay

d_nodal = d_proc + d_queue + d_trans + d_prop d_proc = Processing delay (~microseconds; check bits, find output link) d_queue = Queueing delay (varies; depends on congestion at router) d_trans = Transmission delay = L/R (L=packet length bits, R=link rate bps) d_prop = Propagation delay = d/s (d=link length m, s≈2×10⁸ m/s)
Processing DelayTime to examine header, look up forwarding table. Typically <1 ms. Usually negligible but given explicitly in exam problems.
Queueing DelayTime packet waits in output queue. Depends on traffic intensity La/R. If La/R→1, queue grows unboundedly. Highly variable.
Transmission DelayTime to push all bits onto the link = L/R. For 1500-byte packet on 100 Mbps: (1500×8)/(10⁸) = 0.12 ms.
Propagation DelayTime for signal to travel from sender to receiver = d/s. For 1000 km at 2×10⁸ m/s: 1000×10³/2×10⁸ = 5 ms.

Traffic Intensity and Queueing Delay

Traffic Intensity = La/R where: L = average packet length (bits) a = average packet arrival rate (packets/sec) R = link transmission rate (bps) If La/R ≈ 0: small queueing delay If La/R → 1: very large queueing delay If La/R > 1: queue grows without bound → average delay = infinite!
💡 Key Intuition
Transmission delay is about the SIZE of the packet and the SPEED of the link — how long to push all bits onto the wire. Propagation delay is about the DISTANCE and the speed of light — how long for a bit to travel from A to B once it's on the wire. They are completely independent.

Throughput

Throughput is the rate (bits/time unit) at which bits are transferred between sender and receiver. The bottleneck link is the link on the end-to-end path with the lowest transmission rate — it constrains throughput.

End-to-end throughput = min(R_c, R_s, R/N) R_c = access link rate at client R_s = access link rate at server R = shared backbone link rate N = number of connections sharing backbone In practice: R_c or R_s is usually the bottleneck (last-mile problem)
★ Exam: End-to-End Delay Calculation [Fall 2024 Q2a]
  • Always break down into: d_trans per link + d_prop per link + d_proc at intermediate nodes
  • For N links (N-1 routers): d_trans × N + sum of all d_prop + (N-1) × d_proc
  • Example: 2500-byte packet, 2 Mbps links, 2.5×10⁸ m/s prop, 5 ms proc, links = 5000+4000+1000 km
    d_trans = (2500×8)/(2×10⁶) = 10 ms per link → 3×10 = 30 ms
    d_prop = 20+16+4 = 40 ms (5000→20ms, 4000→16ms, 1000→4ms)
    d_proc = 2×5 = 10 ms (two intermediate routers)
    Total ≈ 80 ms

Protocol Layers and Service Models

Networks are complex. Layering is the key design principle — each layer provides services to the layer above using services from the layer below. This modularizes the system, making it easier to maintain, update, and reason about.

📌 OSI vs Internet Stack
The OSI model has 7 layers — it adds Presentation (handles encryption, compression, data format translation) and Session (dialog control, synchronization) between Application and Transport. The Internet stack omits these because they can be handled at the application level when needed.

Encapsulation

As data travels down the stack at the sender, each layer adds its own header (and sometimes trailer). At the receiver, each layer strips its header and passes the payload up. A message becomes a segment (+ transport header) → datagram (+ network header) → frame (+ link header + trailer).

This is why when you send an HTTP message, it's actually wrapped inside a TCP segment, inside an IP datagram, inside an Ethernet frame. Routers only look at the IP header (layer 3); switches only look at the Ethernet header (layer 2).

Network Security Threats

The Internet was not originally designed with security in mind — early designers assumed "a group of mutually trusting users." Security has been bolted on since.

MalwareSoftware that takes over hosts. Virus: self-replicating, requires user action (e.g., opening attachment). Worm: self-replicating without user action. Trojan horse: disguised as legitimate. Ransomware: encrypts files for ransom. Spyware: records keystrokes, browsing.
DoS / DDoSDenial of Service: overwhelm server/network with bogus traffic. DDoS: Distributed — attacker uses a botnet (many compromised hosts) to flood target simultaneously. Very hard to defend.
Packet SniffingOn broadcast media (shared Ethernet, WiFi), a promiscuous NIC can read ALL passing frames, not just those addressed to it. Wireshark is a legitimate packet sniffer tool.
IP SpoofingSending packets with a false source IP address. Can bypass IP-based filters. Used in DoS attacks and to impersonate others.
PhishingFake emails/websites that look legitimate (e.g., bank login page) to steal credentials. Social engineering attack targeting humans, not systems.
Man-in-the-MiddleAttacker intercepts and potentially modifies communication between two parties. Both parties think they're communicating directly with each other.
CH 02

Application Layer

Network Application Architectures

Client-Server Architecture
  • Always-on server with permanent IP, often in data centers
  • Clients contact server; intermittently connected, can have dynamic IPs
  • Clients don't communicate directly with each other
  • Examples: HTTP (web), SMTP (email), FTP
  • Scalability challenge: single server becomes bottleneck
Peer-to-Peer (P2P) Architecture
  • No always-on server. Peers communicate directly with each other
  • Each peer is both a client and a server
  • Self-scalable: each new peer brings both demand AND capacity
  • Peers are intermittently connected with changing IPs
  • Example: BitTorrent file sharing
  • Challenge: complex management, security concerns

Processes, Sockets, and Addressing

An application consists of pairs of communicating processes (programs running on hosts). Processes send/receive messages through a socket — the interface between the application layer and the transport layer (like a door: the app pushes messages out, the transport layer delivers them).

To identify a process on a host, you need both the IP address (identifies the host, 32 bits) and the port number (identifies which process on that host, 16 bits). Example: HTTP server at 128.119.245.12, port 80.

Data IntegritySome apps (file transfer, web) require 100% reliable delivery. Others (audio streaming) can tolerate some loss.
ThroughputMultimedia apps need minimum guaranteed throughput. "Elastic" apps use whatever they get (email, file transfer).
TimingInteractive apps (VoIP, online games) need low delay (<100ms for good experience). File transfer doesn't care.
SecuritySome apps need encryption (banking, shopping). TLS (Transport Layer Security) provides encrypted TCP.

HTTP — HyperText Transfer Protocol

HTTP is the Web's application-layer protocol. Client (browser) requests objects, server responds. Uses TCP port 80. HTTP is stateless — server maintains no information about past client requests. This keeps servers simple, but means each request is independent.

Non-Persistent vs Persistent HTTP

Non-Persistent HTTP (1.0)
  • One TCP connection per object (open, request, receive, close)
  • Time per object = 2 RTT + transmission time
    (1 RTT: TCP setup; 1 RTT: HTTP request/response)
  • For 10 images + base HTML: 11 connections, 22 RTTs (plus transmission)
  • OS overhead for each TCP connection open/close
Persistent HTTP (1.1)
  • Single TCP connection for all objects from same server
  • Server leaves connection open after response
  • Without pipelining: ~1 RTT per object after connection setup
  • With pipelining: all requests sent back-to-back, responses received in order
  • Much lower overall latency

Non-persistent (no parallel connections): Total = 38 × (2R + T) // 38 objects × (TCP setup + HTTP req/resp + tx time) Non-persistent (with parallel connections, k parallel): Total ≈ 2R + T + ceil(38/k) × (2R + T) Persistent with pipelining: Total = 2R + T_base + R + T_image_1 + ... ≈ 2R + T_total // Initial TCP setup, then all objects pipelined

HTTP Request Message

HTTP messages are in ASCII (human-readable) format. A request has: request line (method, URL, version), header lines, blank line, body.

HEADLike GET but server only returns headers, not the object body. Used to check if object changed without downloading it.
PUTUploads/replaces a file at specified URL on server. Replaces whatever was there.

Cookies — Maintaining State

Since HTTP is stateless, cookies are used to maintain state across requests. When a client first visits a site, the server sends a Set-Cookie header with a unique ID. The browser stores this and sends it back in every subsequent request to that site. The server uses the cookie ID to look up state in its backend database (shopping cart, login status, etc.).

Privacy concern: Third-party cookies (from ad networks like AdX.com embedded in pages) can track your browsing across multiple unrelated websites. Modern browsers (Firefox, Safari) block these by default.

Web Caching (Proxy Server)

A web cache (proxy) stores copies of recently requested objects. When a browser requests an object, it first goes to the cache. If there (cache hit), returned immediately. If not (cache miss), cache fetches from origin, stores a copy, returns to client. The cache acts as both a client (to origin server) and a server (to requesting browser).

Why caching matters: Reduces access link utilization (less traffic leaving institution), reduces response time (cache is closer than origin server), and reduces load on origin servers.

Conditional GET

To ensure cached content is fresh, the browser includes If-Modified-Since: <date> in its request. If the object hasn't changed, the server returns 304 Not Modified (no body) — fast and bandwidth-efficient. If it has changed, the server returns 200 OK with the new content.

HTTP/2 and HTTP/3

HTTP/2 addressed Head-of-Line (HOL) blocking — in HTTP/1.1, if a large object is requested first, small objects behind it in the pipeline must wait. HTTP/2 divides objects into frames and interleaves them, so small objects can be delivered quickly even when a large object is being transferred. It also allows server push (send objects client hasn't asked for yet).

HTTP/3 runs over QUIC (Quick UDP Internet Connections) instead of TCP. QUIC is built on top of UDP and re-implements reliability, congestion control, and security (TLS 1.3) in one layer. Benefits: 0-RTT or 1-RTT connection establishment (vs 3+ RTTs for TCP+TLS), no head-of-line blocking at transport level, better multiplexing.

Email: SMTP, IMAP, POP3

The email system has three main components: User agents (mail clients like Outlook), mail servers (store mailboxes, maintain outgoing message queues), and SMTP (the protocol for transferring mail between servers).

SMTP — Simple Mail Transfer Protocol

SMTP uses TCP port 25. It's a client push protocol — the sending server initiates the TCP connection and pushes the message to the receiving server. Phases: handshaking, transfer, closure. Commands and responses are ASCII text.

S: 220 hamburger.edu C: HELO crepes.fr S: 250 Hello crepes.fr, pleased to meet you C: MAIL FROM: <alice@crepes.fr> S: 250 alice@crepes.fr... Sender ok C: RCPT TO: <bob@hamburger.edu> S: 250 bob@hamburger.edu... Recipient ok C: DATA S: 354 Enter mail, end with "." on a line by itself C: Do you like ketchup? C: . S: 250 Message accepted for delivery C: QUIT
SMTP vs HTTP Comparison

Both use ASCII command/response interactions with status codes. But SMTP is push (sender initiates), HTTP is pull (client requests). SMTP requires message body to be 7-bit ASCII. SMTP can send multiple objects in one message (MIME for attachments). HTTP encapsulates each object separately.

Mail Access Protocols

  • Synchronizes across multiple devices
  • Used by Gmail, corporate email

DNS — Domain Name System

DNS is a distributed hierarchical database that translates human-readable hostnames (www.aust.edu.bd) to IP addresses (203.190.10.1). It's implemented as an application-layer protocol where hosts and servers exchange DNS messages to resolve names. DNS is critical — almost every Internet transaction starts with a DNS lookup.

Why Not Centralize DNS?

A single DNS server would be: a single point of failure, overwhelmed with traffic volume (Cloudflare's DNS does 1 trillion queries/day), far from many users (high delay), and a maintenance nightmare. Instead, DNS is distributed and hierarchical.

DNS Hierarchy

Root DNS servers: 13 logical root servers (actually hundreds of physical machines). "Last resort" — contacted when local DNS can't resolve. ICANN manages the root zone.

TLD DNS servers: Responsible for top-level domains (.com, .org, .edu, .bd). Network Solutions manages .com, .net. Educause manages .edu.

Authoritative DNS servers: The organization's own server providing authoritative hostname→IP mappings. E.g., aust.edu.bd's authoritative server knows all hostnames in that domain.

Local DNS server: Each ISP has one. First contact for DNS queries. Caches recent answers. Doesn't strictly belong to hierarchy but acts as proxy.

Iterative vs Recursive Queries

Iterative Query
  • Local DNS server does all the work
  • Contacts root → root says "ask .com TLD"
  • Contacts TLD → TLD says "ask amazon.com authoritative"
  • Contacts authoritative → gets IP address
  • Less burden on upper-level servers (preferred in practice)
Recursive Query
  • Each server contacts the next on behalf of the requester
  • Local asks root; root asks TLD; TLD asks authoritative
  • Result passed back through the chain
  • Puts heavy load on upper-level servers (less common)

DNS Caching

Once a DNS server learns a mapping, it caches it and returns cached responses immediately (much faster). Cached entries expire after TTL (Time to Live). TLD server addresses are typically cached in local DNS servers, allowing root server bypass. Drawback: if a host changes its IP, the change may not propagate globally until all TTLs expire (stale cache problem).

Video Streaming and CDNs

Video streaming is the dominant internet traffic. The challenge is delivering high-quality video to millions of simultaneous users with heterogeneous network conditions.

DASH — Dynamic Adaptive Streaming over HTTP

Server divides video into chunks, encodes each at multiple bit rates, stores them in different files. Client periodically estimates available bandwidth and chooses the highest encoding rate sustainable. Can switch quality mid-stream as network conditions change. The "intelligence" is at the client.

Content Distribution Networks (CDNs)

Rather than serving from one mega-server (which creates a single point of failure and is far from most users), CDNs deploy servers at hundreds of locations worldwide. Two strategies: "Enter deep" — push servers deep into access networks, very close to users (Akamai: 240,000+ servers). "Bring home" — smaller number of large clusters near access networks (Limelight). When a user requests content, they're directed to a nearby CDN server.

CH 03

Transport Layer

Transport Layer Overview

The transport layer provides logical communication between processes (not hosts — that's the network layer). It sits between the application layer (processes) and the network layer (hosts). The network layer delivers packets between hosts; the transport layer extends this to deliver between the right processes on those hosts using ports.

TCP — Transmission Control Protocol
  • Reliable, in-order byte stream delivery
  • Connection-oriented (3-way handshake required)
  • Flow control: sender won't overflow receiver
  • Congestion control: throttles when network congested
  • Full-duplex: data flows in both directions simultaneously
UDP — User Datagram Protocol
  • Unreliable, unordered delivery ("best effort")
  • Connectionless — no setup required
  • No flow control, no congestion control
  • Small header (8 bytes vs TCP's 20+)
  • Lower latency; used for real-time apps

Multiplexing and Demultiplexing

Multiplexing (at sender): gathering data chunks from multiple sockets, adding transport header info, passing segments to network layer. Demultiplexing (at receiver): using header fields to deliver received segments to the correct socket (correct application process).

UDP Demultiplexing (Connectionless)
  • Based on destination port number only
  • Datagrams with same dest port → same UDP socket, regardless of source IP or source port
  • Example: DNS server at port 53 receives queries from many clients; all go to the same socket
TCP Demultiplexing (Connection-Oriented)
  • Based on 4-tuple: (src IP, src port, dst IP, dst port)
  • Two segments with same dst IP:port but different src IP:port → different sockets
  • Web server on port 80 can have thousands of simultaneous connections, each its own socket

UDP — User Datagram Protocol

UDP provides exactly what IP provides plus multiplexing/demultiplexing (port numbers) and an optional checksum. It's "no frills" — segments may be lost, delivered out of order. Why use it?

  • No connection setup: saves the 1–1.5 RTT of TCP's handshake
  • No connection state: simpler, less memory per client (server can handle more connections)
  • Small header size: 8 bytes vs TCP's 20+ bytes — less overhead for small messages
  • No congestion control: can transmit as fast as application wants (useful for real-time)

Used for: DNS (quick lookups), streaming media (loss tolerable), SNMP (management), HTTP/3.

UDP Segment Header (8 bytes)

| Source Port (16 bits) | Destination Port (16 bits) | | Length (16 bits) | Checksum (16 bits) | | Data ... |

UDP Checksum

The checksum detects bit errors in the transmitted segment. Sender treats contents as 16-bit integers, adds them with 1's complement arithmetic, puts result in checksum field. Receiver performs the same sum; if result is all 1s, no error detected. Note: the checksum is weak — it can miss some errors (two flipped bits in the same column cancel out).

Reliable Data Transfer (RDT) — Building Reliable Protocols

The underlying network (IP) is unreliable — it can corrupt, lose, or reorder packets. The transport layer must build reliability on top. RDT is studied incrementally, adding mechanisms for each type of unreliability.

rdt 1.0 — Perfectly Reliable Channel

Assume no errors, no loss. Simple FSM: sender sends when called from above; receiver delivers when packet arrives. No special mechanisms needed. (Not realistic, but a starting point.)

rdt 2.0 — Bit Errors (Channel Can Flip Bits)

Add checksums to detect errors. Receiver sends ACK (acknowledgement, "got it correctly") or NAK (negative acknowledgement, "error, resend"). Sender retransmits on NAK. Uses stop-and-wait: send one packet, wait for response before sending next.

Fatal Flaw in rdt2.0
What if the ACK or NAK itself is corrupted? Sender doesn't know if receiver got the packet. Can't just retransmit (receiver might get a duplicate and think it's new data).

rdt 2.1 — Handling Corrupted ACK/NAKs

Solution: add sequence numbers to packets. When sender gets corrupted ACK/NAK, it retransmits. Receiver checks sequence number to detect duplicates and discard them. Only 2 sequence numbers needed (0 and 1) for stop-and-wait.

rdt 2.2 — NAK-Free (ACKs Only)

Instead of sending NAK, receiver sends ACK for the last correctly received packet with its sequence number. A duplicate ACK at the sender means the packet wasn't received correctly — same action as NAK. This is what TCP does.

rdt 3.0 — Packet Loss (Adds Timer)

Now the channel can also lose packets. Add a countdown timer at sender. If no ACK received within timeout period, retransmit. Handles both packet loss and ACK loss (duplicate packets handled by sequence numbers already).

rdt 3.0 Performance Problem
Stop-and-wait means only one packet in flight at a time. Sender utilization = (L/R) / (RTT + L/R) Example: L=8000 bits, R=10⁹ bps (1 Gbps), RTT=30ms: d_trans = 8000/10⁹ = 0.008 ms = 8 μs U_sender = 0.008ms / (30ms + 0.008ms) ≈ 0.00027 = 0.027% Only 0.027% utilization on a 1 Gbps link! The rest of the time the sender is idle, waiting for ACK. Solution: pipelining — keep multiple packets in flight.

Pipelined Reliable Transfer: GBN and SR

Pipelining allows multiple unACKed packets to be in flight simultaneously. Range of sequence numbers must be increased. Two main approaches:

Go-Back-N (GBN)

  • Sender window of size N: up to N unACKed packets in flight simultaneously
  • Cumulative ACK: ACK(n) acknowledges all packets up through and including seq # n
  • Single timer for oldest unACKed packet
  • On timeout(n): retransmit packet n AND ALL higher seq # packets in window — "go back" to n
  • Receiver: ACK-only (no buffering of out-of-order). Out-of-order packets are discarded; re-ACK highest in-order received
  • Window size constraint: ≤ 2^k - 1 (k = # seq # bits)

Selective Repeat (SR)

  • Sender window of size N: up to N unACKed packets in flight
  • Individual ACK per packet — receiver ACKs each correctly received packet separately
  • Separate timer per unACKed packet
  • On timeout(n): retransmit ONLY packet n (hence "selective")
  • Receiver buffers out-of-order packets while waiting for missing earlier packets
  • Window size constraint: ≤ 2^k / 2 (must be ≤ half the seq # space to avoid ambiguity)

TCP Reliable Data Transfer

TCP builds reliable transfer on top of IP using: checksums (detect corruption), sequence numbers (detect loss and duplicates), ACKs (confirm delivery), and timers (trigger retransmission on loss).

Fast Retransmit: TCP doesn't always wait for timeout. If sender receives 3 duplicate ACKs (same ACK number 4 times total), it infers the packet following the ACK was lost and retransmits immediately — before the timer fires. This dramatically speeds up recovery from loss.

TCP ACK generation rules (RFC 5681): 1. Segment arrives in order, no gap, previous data ACKed → Wait up to 500ms for next segment, then send ACK 2. Segment arrives in order, one ACK pending → Immediately send cumulative ACK for both segments 3. Out-of-order segment (gap detected) → Immediately send DUPLICATE ACK indicating seq# expected 4. Segment fills gap → Immediately send ACK if segment starts at lower end of gap

TCP Flow Control

Flow control prevents the sender from overwhelming the receiver's buffer. The receiver advertises its remaining free buffer space (rwnd) in the TCP header of every segment sent back to the sender. The sender ensures the amount of unACKed in-flight data never exceeds rwnd.

Receiver: sets rwnd = RcvBuffer - [LastByteRcvd - LastByteRead] Sender: ensures LastByteSent - LastByteACKed ≤ rwnd If rwnd = 0: sender still sends 1-byte segments to probe (prevents deadlock where receiver has freed space but sender doesn't know)

TCP 3-Way Handshake

Before any data exchange, TCP must establish a connection. This synchronizes sequence numbers and allocates buffers on both sides.

Client → Server: SYN segment. SYNbit=1, Seq=client_isn (randomly chosen). Client enters SYN_SENT state.

Server → Client: SYNACK segment. SYNbit=1, Seq=server_isn (server's random ISN), ACKbit=1, ACKnum=client_isn+1. Server allocates buffers. Enters SYN_RCVD state.

Client → Server: ACK segment. ACKbit=1, ACKnum=server_isn+1. Client enters ESTABLISHED state. Can contain data. Server enters ESTABLISHED on receipt.

Why 3 ways? 2-way handshake can fail: a retransmitted SYN from an old connection could establish a half-open connection. The 3-way handshake ensures both sides are alive and ready before exchanging data.

TCP Connection Teardown

Client closes socket → sends FIN segment (FINbit=1). Client enters FIN_WAIT_1.

Server receives FIN → sends ACK. Server can still send data (half-closed). Client enters FIN_WAIT_2.

Server finishes sending → sends FIN. Server enters LAST_ACK.

Client receives FIN → sends ACK. Client enters TIMED_WAIT (waits 2×max segment lifetime ≈ 30–120 seconds before fully closing). Server enters CLOSED on receipt.

Congestion Control

Why Congestion Happens and Its Costs

Congestion occurs when too many sources send data too fast for the network to handle. It manifests as large queuing delays and packet loss at routers. Key insights from congestion cost analysis:

  • Throughput can never exceed capacity
  • Delay increases as capacity is approached
  • Loss/retransmission decreases effective throughput
  • Premature timeouts cause unnecessary duplicates, further reducing throughput
  • Packets lost downstream waste all upstream capacity used to carry them
End-End Congestion Control
  • No explicit feedback from network
  • Congestion inferred from observed loss and/or delay
  • This is what TCP uses
  • Routers don't need to do anything special
Network-Assisted Congestion Control
  • Routers explicitly signal congestion to hosts
  • Single-bit indicator (ECN — Explicit Congestion Notification)
  • Or router explicitly sets sending rate
  • Used in ATM, DECbit, TCP ECN extension

TCP AIMD Congestion Control

TCP adjusts its sending rate by maintaining a congestion window (cwnd). The sending rate ≈ cwnd/RTT bytes/sec. The strategy is Additive Increase, Multiplicative Decrease (AIMD): slowly probe for unused bandwidth, then back off quickly when congestion detected.

AIMD Rules: • No loss: increase cwnd by 1 MSS per RTT (additive increase) • Loss by triple dup ACK: cwnd = cwnd/2 (multiplicative decrease, Reno) • Loss by timeout: cwnd = 1 MSS (more severe response, both Tahoe and Reno) TCP sending constraint: LastByteSent - LastByteACKed ≤ min(cwnd, rwnd)

TCP Slow Start

At connection start, cwnd = 1 MSS. Then double cwnd every RTT (increment by 1 MSS for every ACK received). This sounds slow but grows exponentially — doubles every round-trip. Continues until cwnd reaches ssthresh (slow start threshold) or loss occurs.

Congestion Avoidance

When cwnd ≥ ssthresh: increase cwnd by 1 MSS per RTT (linear growth). This is the "probe" phase — slowly increasing to find the bottleneck. One MSS per RTT = 1/cwnd MSS per ACK (increment = MSS × MSS/cwnd on each ACK).

Network Layer: Data Plane + Control Plane

Network Layer Overview

The network layer is the lowest layer that deals with end-to-end communication across multiple links. Every host and router has a network layer. Two key functions: forwarding (local, nanosecond-scale, hardware) and routing (network-wide, millisecond-scale, software).

Data Plane (Per-Router, Local)
  • Determines how packet arriving at router's input port is forwarded to output port
  • Uses local forwarding table (computed by control plane)
  • Hardware implementation, nanosecond timescale
  • Traditional forwarding: based only on destination IP address
  • Generalized forwarding: based on any header field (OpenFlow)
Control Plane (Network-Wide)
  • Determines how packet is routed from source to destination across multiple routers
  • Computes forwarding tables
  • Two approaches: per-router (traditional OSPF/BGP) or logically centralized (SDN)
  • Software, millisecond timescale

Router Architecture

A router consists of four components:

Input PortsPhysical termination (bits → link layer frames). Lookup forwarding table to determine output port. Buffers packets arriving faster than switching fabric can handle. Site of HOL blocking.
Switching FabricTransfers packets from input to output port. Three types: memory (first-gen), bus (limited by bus bandwidth, ~32 Gbps), interconnection network/crossbar (scalable, most modern).
Output PortsBuffers and transmits packets. Scheduling discipline determines which queued packet to send next. HOL blocking can also occur at output port when fabric is fast and output port is congested.
Routing ProcessorRuns routing algorithms/protocols (OSPF, BGP). Computes forwarding tables. Control plane software.

Head-of-Line (HOL) Blocking

HOL blocking occurs at input ports when multiple packets at the front of different input queues want to go to the same output port simultaneously — only one can proceed, leaving the others blocked. Even packets behind the blocked ones, destined for FREE output ports, cannot advance because the first packet in their queue is stuck. This reduces effective throughput.

★ Exam [Fall 2024 Q1e]
In a router with N input ports each connected to a switching fabric: If port 1 has packets for output port A and output port B queued, and port 2 also has a packet for output port A, then port 1's packet for A wins. Port 2's packet for A waits. But also, port 1's packet for output B (which is free) is blocked because it's behind the port-A packet in port 1's queue. That's HOL blocking.

DHCP — Dynamic Host Configuration Protocol

DHCP allows a host to automatically obtain an IP address when it connects to a network — "plug-and-play." DHCP server (usually co-located in router) assigns addresses from a pool, along with subnet mask, default gateway IP, and DNS server IP.

DHCP Discover: Client broadcasts (src=0.0.0.0, dst=255.255.255.255) "Is there a DHCP server?"

DHCP Offer: Server broadcasts an offer with an IP address, lease duration, subnet mask, gateway, DNS.

DHCP Request: Client broadcasts "I'd like to use that IP address" (there may be multiple servers, this tells them which offer was accepted).

DHCP ACK: Server confirms. Client now has a valid IP for the lease duration.

Why broadcast?
The client doesn't have an IP yet, so it can't do unicast. The server's reply is also broadcast because the client doesn't have an IP to receive a unicast packet. Everything is encapsulated in UDP/IP/Ethernet, sent to broadcast addresses.

NAT — Network Address Translation

NAT allows all devices in a local network (using private IPs) to share a single public IP address. The NAT router maintains a translation table.

Private IP ranges (RFC 1918 — not routable on public Internet): 10.0.0.0/8 (10.x.x.x — large networks) 172.16.0.0/12 (172.16.x.x – 172.31.x.x) 192.168.0.0/16 (192.168.x.x — home/small networks) NAT Translation Table entry: (LAN IP, LAN port) ↔ (WAN IP, WAN port) Outgoing: replace src=(192.168.1.4:3345) with (138.76.29.7:5001) Incoming: replace dst=(138.76.29.7:5001) with (192.168.1.4:3345)

Advantages of NAT: conserves IPv4 address space; provides security (internal hosts not directly addressable from outside); can change ISP or internal addresses without notifying outside world. Controversy: violates end-to-end principle (port manipulation at network layer); what if external client wants to connect to internal server?

IPv6

IPv4 has 32-bit addresses — ICANN allocated the last block in 2011. IPv6 uses 128-bit addresses (2¹²⁸ — essentially unlimited). Other improvements: fixed 40-byte header (no fragmentation at routers, no checksum — faster processing), flow labels for QoS.

  • No header checksum (reduces processing per hop)
  • No options (moved to "next header" extensions)
IPv4-to-IPv6 Transition
  • Dual stack: nodes run both IPv4 and IPv6

Control Plane: Routing Algorithms

Routing algorithms determine the "good path" (typically least-cost path) from source to destination through the network graph. Graph abstraction: nodes = routers, edges = links, edge weights = link costs (configured by network operators: can be 1, bandwidth-inverse, congestion-based, etc.)

Algorithm Classification

Link State (Global)
  • All routers know complete topology
  • Each router broadcasts its link costs to all others (link state broadcast)
  • Each router runs Dijkstra locally to compute shortest paths
  • O(n²) time, O(n²) messages (n = #nodes)
  • Used in: OSPF, IS-IS
Distance Vector (Decentralized)
  • Each router knows only costs to its direct neighbors
  • Iteratively exchanges distance vectors with neighbors
  • Based on Bellman-Ford equation
  • Distributed, asynchronous, self-stopping
  • Used in: RIP, BGP (path vector)

Dijkstra's Link-State Algorithm

Notation: N' = set of nodes whose least-cost path is definitively known D(v) = current cost estimate from source to v p(v) = predecessor node on path from source to v c(x,y) = direct link cost from x to y (∞ if not neighbors) Algorithm: Initialization: N' = {u} (u = source) for all nodes v: if v is adjacent to u: D(v) = c(u,v), p(v) = u else: D(v) = ∞ Loop until N' contains all nodes: 1. Find w ∉ N' with minimum D(w) → add w to N' 2. For all v adjacent to w, v ∉ N': D(v) = min(D(v), D(w) + c(w,v)) if updated: p(v) = w

Distance Vector Algorithm

Bellman-Ford equation: D_x(y) = min_v { c(x,v) + D_v(y) } (best path from x to y = min over all neighbors v of {cost x-to-v + v's best to y}) Algorithm: Each node x: maintains D_x(y) for all y maintains c(x,v) for each neighbor v knows neighbor v's distance vector D_v(y) When x receives updated DV from neighbor v: for each destination y: D_x(y) = min_v { c(x,v) + D_v(y) } if any D_x(y) changed, send updated DV to all neighbors Converges when no more updates.

Count-to-Infinity Problem

Distance vector has a "good news travels fast, bad news travels slow" problem. When a link cost decreases, the update propagates quickly. When a link cost increases (or a link fails), the algorithm may loop for many iterations before converging.

Example: x-y-z, c(x,y) increases from 4 to 60 Initial: D_y(x)=4, D_z(x)=5 (z thinks it reaches x via y for cost 5) y detects c(x,y)=60. Now computes D_y(x) = min{60, 1+5} = 6 via z z sees D_y(x)=6, computes D_z(x) = min{∞, 1+6} = 7 via y y sees D_z(x)=7, computes D_y(x) = min{60, 1+7} = 8 via z ... counts all the way up to 50 (actual cost = 60) Solutions: Split horizon: don't advertise a route back to the node you learned it from Poison reverse: advertise the route but with infinite cost back to the source

Intra-AS Routing: OSPF

The Internet is divided into Autonomous Systems (AS) — groups of routers under common administrative control. Routing within an AS is intra-AS routing; between ASes is inter-AS routing.

OSPF (Open Shortest Path First) is the most common intra-AS protocol. It uses link-state routing with Dijkstra's algorithm. Each router floods its link-state information directly to all routers in the AS using IP (not TCP/UDP). Each router has the full topology. Multiple link cost metrics possible (bandwidth, delay). All OSPF messages are authenticated.

OSPF supports a two-level hierarchy: local areas and a backbone. Area border routers summarize area topology for the backbone. This reduces flooding overhead in large ASes.

Inter-AS Routing: BGP

BGP (Border Gateway Protocol) is the de facto inter-domain routing protocol — "the glue that holds the Internet together." BGP allows each AS to: (1) advertise which IP prefixes it can reach; (2) determine routes to other networks based on reachability and policy.

eBGP (External BGP)
  • Between gateway routers of different ASes
  • Advertises reachability to external prefixes
  • "AS3 can reach prefix X"
iBGP (Internal BGP)
  • Between routers within same AS
  • Propagates externally-learned routes internally
  • All internal routers learn about external destinations

BGP advertises routes, not just distances. A route = prefix + attributes. Key attributes: AS-PATH (list of ASes the advertisement has passed through), NEXT-HOP (first router outside AS on the path). Policy-based: ASes can decide which routes to accept/reject/advertise based on business relationships (avoid transit for competitors, prefer customer routes, etc.).

Hot potato routing: when a router inside an AS can reach an external destination via multiple exit points, it chooses the exit point with the least intra-AS cost — "dump the hot potato quickly." Doesn't optimize the overall path, just minimizes cost within own AS.

Error Detection Techniques

Single-Bit Parity

Add one parity bit so total number of 1s in the data+parity is even (even parity) or odd (odd parity). Can detect but not correct single-bit errors. Even number of bit flips will go undetected.

Two-Dimensional Parity

Arrange bits in a matrix. Compute parity for each row AND each column. Can detect and correct single-bit errors — the error is at the intersection of the faulty row and faulty column. Can detect but not correct 2-bit errors.

Internet Checksum

Used in TCP, UDP, IP. Treat segment contents as 16-bit integers, add them with 1's complement arithmetic, invert the result. Receiver adds all content + checksum; if all 1s, no error detected. Fast to compute, weak protection (two canceling flips undetected).

CRC — Cyclic Redundancy Check

More powerful than parity or checksum. Based on polynomial division in GF(2) (arithmetic using XOR instead of addition). Used in Ethernet, 802.11 WiFi.

Given: D = data bits, G = generator polynomial (r+1 bits), r = degree of G Step 1: Append r zeros to D → call it D·2^r Step 2: Divide D·2^r by G using XOR long division Step 3: Remainder R has exactly r bits → this is the CRC Step 4: Transmit ⟨D, R⟩ (D concatenated with R) Receiver: divide ⟨D', R'⟩ by G If remainder = 0: no error detected (pass) If remainder ≠ 0: error detected (drop frame) CRC detects all burst errors of length ≤ r bits. CRC-32 (r=32): detects all bursts ≤ 32 bits and most longer ones.

Multiple Access Protocols

When a link is shared (broadcast medium), multiple nodes transmit on the same channel. If two nodes transmit simultaneously, their signals interfere — a collision. A multiple access protocol determines when each node can transmit.

An ideal MAC protocol on a channel of rate R bps: (1) when 1 node wants to transmit, it can at rate R; (2) when M nodes want to transmit, each gets rate R/M on average; (3) fully decentralized (no coordinator, no clock sync); (4) simple.

Channel Partitioning Protocols

TDMA (Time Division MA)
  • Divide time into frames; each frame divided into slots
  • Each node assigned a slot per frame
  • Pros: no collisions, fair, equal bandwidth
  • Cons: slot goes idle if node has nothing to send; all nodes must wait for their slot even if others are silent
FDMA (Frequency Division MA)
  • Divide channel spectrum into frequency bands
  • Each node assigned a fixed frequency band
  • Pros: no collisions, fair
  • Cons: frequency band idle if node silent; wasted capacity

Random Access Protocols

No advance coordination. When a node has a frame to send, it transmits at full rate R. If collision: wait random time, retransmit. "Recover from" collisions rather than avoid them.

Pure ALOHATransmit immediately when frame ready. Collision = overlap with any frame sent in [t₀-1, t₀+1]. Efficiency = 1/(2e) ≈ 18%. Simpler but worse than slotted.
Slotted ALOHATime divided into slots. Only transmit at start of slot. Collision = same slot. Efficiency = 1/e ≈ 37%. Requires clock synchronization.
CSMA (Carrier Sense MA)"Listen before talking." If channel sensed idle, transmit. If busy, defer. But propagation delay means two nodes may both sense idle and collide (they can't hear each other's just-started transmissions).
CSMA/CD (Collision Detection)While transmitting, listen for collision. If collision detected, abort immediately and send jam signal. Then binary exponential backoff. Used in Ethernet. Reduces wasted time vs pure CSMA.

Ethernet CSMA/CD Algorithm

Get datagram from network layer, create frame

Sense channel: if idle → start transmitting; if busy → wait until idle, then transmit

If entire frame transmitted without collision → done!

If collision detected during transmission → abort, send jam signal (ensures all nodes detect collision)

Binary exponential backoff: after m-th collision, choose K randomly from {0,1,2,...,2^m-1}. Wait K×512 bit times. Return to step 2. (More collisions = longer possible wait = less likely to collide again)

"Taking Turns" Protocols

Polling
  • Central master node polls each slave node in round-robin order
  • Slave can only transmit when polled
  • Pros: no collisions, can give more turns to busier nodes
  • Cons: polling overhead, latency (must wait for poll), single point of failure (if master fails, network down)
  • Used in: Bluetooth (master-slave)
Token Passing
  • A special control token message is passed node-to-node in sequence
  • Node can only transmit while holding the token
  • Pros: no collisions, decentralized (no master), fair
  • Cons: token overhead, latency, single point of failure (lost/corrupted token)
  • Used in: FDDI, Token Ring, IEEE 802.4
★ Exam [Fall 2024 Q1g]
Both polling and token passing are "taking turns" protocols — best of both worlds between channel partitioning (efficient at high load) and random access (efficient at low load). Key disadvantages to know: polling = central master failure; token = token loss. Bluetooth uses polling. FDDI uses token passing.

MAC Addresses and ARP

MAC Addresses

Every NIC has a unique 48-bit MAC address (also called physical address or Ethernet address) burned into its ROM. Written as hex: 1A-2F-BB-76-09-AD. Managed by IEEE — manufacturers buy blocks of addresses. Unlike IP addresses, MAC addresses are flat (not hierarchical) and portable (don't change when you move to a different network, because they don't encode location).

MAC Address
  • Like a Social Security Number — uniquely identifies a NIC
  • Flat — no structure encoding location
  • Portable: same address wherever you go
  • Burned in NIC ROM (though can be software-set)
IP Address
  • Like a postal address — encodes location in network
  • Hierarchical — network part + host part
  • Changes when you move to different subnet
  • Assigned by DHCP or manually

ARP — Address Resolution Protocol

When a node wants to send a frame to a destination IP, it needs the destination's MAC address. ARP resolves IP → MAC within the same subnet.

A wants to send to B (same subnet). B's MAC not in A's ARP table.

A broadcasts ARP query: "Who has IP 137.196.7.14? Tell 137.196.7.23." Destination MAC = FF:FF:FF:FF:FF:FF (broadcast).

All nodes on LAN receive the query. Only B (with IP 137.196.7.14) responds: "I have it. My MAC is 58-23-D7-FA-20-B0."

A caches the entry (IP: 137.196.7.14, MAC: 58-23-D7-FA-20-B0, TTL≈20min). Now sends frame directly to B's MAC.

ARP Across Different Subnets

A cannot directly ARP for B if B is in a different subnet. Instead, A sends to the default gateway router:

A wants to send to B (different subnet 222.x). A looks up B's IP in its routing table → use default gateway R.

A ARPs for R's MAC (R is on A's subnet). A sends frame: IP src=A, IP dst=B, MAC src=A's MAC, MAC dst = R's MAC.

R receives frame, strips Ethernet header, reads IP datagram. Looks up B in routing table → R knows which interface to use to reach B's subnet.

R ARPs for B's MAC on B's subnet. R sends NEW frame: IP src=A, IP dst=B (unchanged!), MAC src=R's MAC on B-side, MAC dst = B's MAC.

B receives and processes. Key rule: IP addresses never change; MAC addresses change at every hop.

This is one of the most comprehensive exam questions — it tests understanding of multiple layers simultaneously. Build ARP tables for each host/router at each step. Remember: A needs R's MAC to even send the first packet. R needs B's MAC to forward. Each router interface has its own MAC and IP.

Ethernet

Ethernet is the dominant wired LAN technology. First widely used LAN protocol. Simple, cheap, and has scaled from 10 Mbps to 400 Gbps. Uses CSMA/CD as its MAC protocol.

Ethernet Frame Structure

| Preamble (7B) | SFD (1B) | Dest MAC (6B) | Src MAC (6B) | Type (2B) | Data (46-1500B) | CRC (4B) | Preamble: 10101010 × 7 bytes — synchronizes receiver/sender clocks SFD: 10101011 — "Start Frame Delimiter," frame is beginning Dest MAC: receiver's 48-bit MAC address (or broadcast FF:FF:FF:FF:FF:FF) Src MAC: sender's 48-bit MAC address Type: identifies higher-layer protocol (0x0800=IP, 0x0806=ARP) Data: payload, 46-1500 bytes (minimum 64 bytes total for CSMA/CD to work) CRC-32: error detection, 4 bytes

Ethernet is connectionless (no handshaking between NICs) and unreliable (receiving NIC doesn't send ACKs). Dropped frames are recovered only by higher-layer protocols (TCP). Many Ethernet standards exist (10BASE-T, 100BASE-TX, 1000BASE-T, etc.) but all share the same MAC protocol and frame format.

Ethernet Switches

A switch is a link-layer device (layer 2). It stores-and-forwards Ethernet frames, selecting the right outgoing interface based on MAC addresses. Unlike hubs (which broadcast everything), switches selectively forward.

Key properties: Transparent (hosts unaware of switch's presence), plug-and-play / self-learning (no configuration needed).

Switch Self-Learning

When frame arrives on interface x with source MAC A, switch records: "A is reachable via interface x" in switch table (with timestamp for TTL).

Index switch table with destination MAC. If entry exists and destination is NOT on the incoming interface → forward frame on that interface.

If destination is on the SAME interface it arrived → drop (no need to forward).

If destination unknown (not in table) → flood: forward on ALL interfaces except the one it arrived on.

Day in the Life of a Web Request

This synthesis ties together every layer. Scenario: a student's laptop connects to campus WiFi and requests www.google.com.

DHCP (Get IP address): Laptop broadcasts DHCP Discover → campus router (running DHCP server) sends DHCP Offer → laptop sends DHCP Request → DHCP ACK. Laptop now has: IP address, subnet mask, default gateway IP, DNS server IP.

ARP (Get gateway MAC): Laptop needs to send DNS query to DNS server, but must first send it to default gateway. ARP broadcast to get gateway's MAC. Gateway replies with its MAC. Now laptop can form frames.

DNS (Get Google's IP): DNS query (UDP, dest=DNS server, port 53) encapsulated in IP→Ethernet→WiFi frame. Forwarded by switches and routers through campus → ISP → global Internet to DNS server. DNS reply: 64.233.169.105.

TCP (Establish connection): Laptop sends TCP SYN to 64.233.169.105:80. Routed across Internet to Google. Google responds with SYNACK. Laptop sends ACK. TCP connection established.

HTTP (Get page): Laptop sends HTTP GET / HTTP/1.1. Google responds with HTML. Each segment wrapped in TCP→IP→Ethernet. Switches forward by MAC; routers forward by IP. HTML displayed in browser.

Network Security

What is Network Security?

Network security is about ensuring that communication between parties is private, authenticated, and unaltered. Alice and Bob want to communicate securely; Trudy is an intruder who may be anywhere on the path between them.

ConfidentialityOnly the intended sender and receiver can understand the message contents. Achieved by encryption. No eavesdropper (even with the ciphertext) can understand the message.
AuthenticationSender and receiver want to confirm each other's identity. "Is this really Alice, or is Trudy impersonating her?"
Message IntegrityEnsure message was not altered in transit (or after), without detection. Even if Trudy can't read the message, she might flip bits.
Operational SecurityProtecting against malware, DoS attacks, packet sniffing, IP spoofing. Firewalls, IDS (Intrusion Detection Systems).

What bad guys can do: Eavesdrop (sniff packets), Actively modify messages (insert/delete), Impersonation (spoof source addresses), Hijacking (take over ongoing connection), Denial of Service (overwhelm with bogus traffic).

Principles of Cryptography

Cryptography is the science of encoding (encrypting) and decoding (decrypting) messages so only intended parties can read them. Cryptanalysis is the art of breaking ciphers without the key.

Terminology: m = plaintext (original message) K_A(m) = ciphertext (encrypted with key K_A) K_B(K_A(m)) = m (decryption with key K_B recovers plaintext) Symmetric key: K_A = K_B (same key for encryption and decryption) Asymmetric key: K_A ≠ K_B (public key encrypts, private key decrypts)

Symmetric Key Cryptography

Sender and receiver share the same secret key K_S. Both encryption and decryption use K_S. The main challenge: how do parties agree on K_S without Trudy intercepting it?

Caesar Cipher (Shift Cipher)

Substitute each letter with the letter k positions later in the alphabet. Key K = shift value. Example with K=3: a→d, b→e, c→f. Very easy to break (only 25 possible keys; try all).

Monoalphabetic Substitution Cipher

Any permutation of the 26 letters can be the key — letter a maps to any other letter (uniquely). 26! ≈ 4×10²⁶ possible keys — brute force is infeasible. But statistical analysis breaks it easily (e.g., 'e' is most common letter in English; frequency analysis of ciphertext reveals 'e' quickly). Known-plaintext attack is also effective.

Plaintext: abcdefghijklmnopqrstuvwxyz Ciphertext: mnbvcxzasdfghjklpoiuytrewq Plaintext message: "bob. i love you. alice" Ciphertext: "nkn. s gktc wky. mgsbc"

Polyalphabetic Encryption

Use multiple monoalphabetic ciphers in a cycling pattern. Each position in the message uses a different substitution cipher. Much harder to break with frequency analysis since the same plaintext letter maps to different ciphertext letters depending on position.

Example: two Caesar ciphers C1 (k=5) and C2 (k=19), pattern C1,C2,C2,C1,C2... Plaintext: b o b, i l o v e y o u. Cipher: C1 C2 C2 C1 C2 C1 C2 C2 C1 C2 C1 Ciphertext: g h u, n e t o x d h z.

Block vs Stream Ciphers

Block Cipher
  • Message encrypted in fixed-size blocks (e.g., 64 or 128 bits)
  • Same key maps same plaintext block → same ciphertext block
  • Examples: DES (56-bit key, now broken), AES (128/192/256-bit key, current standard)
  • Can add modes (CBC, CTR) to handle repeated blocks
Stream Cipher
  • Encrypt bit-by-bit or byte-by-byte
  • Key generates a pseudorandom keystream; XOR with plaintext
  • Faster for continuous data (voice, video)
  • Examples: RC4, ChaCha20

Public Key Cryptography

Public key (asymmetric) cryptography solves the key distribution problem of symmetric key crypto. Each entity has a key pair: a public key K⁺ (shared with everyone) and a private key K⁻ (kept secret). Anyone can encrypt with the public key, but only the owner can decrypt with the private key.

Two crucial properties: (1) K⁻(K⁺(m)) = m — private key decrypts what public key encrypts. (2) K⁺(K⁻(m)) = m — public key also decrypts what private key encrypts (used for digital signatures). Given K⁺, it is computationally infeasible to determine K⁻.

RSA — Rivest, Shamir, Adelson Algorithm

RSA is based on the mathematical difficulty of factoring large numbers. Multiply two large primes → trivial. Given their product, find the original primes → computationally infeasible for large numbers (no known efficient algorithm).

RSA Key Generation

Choose two large prime numbers p and q. (In practice, 1024 bits each.)

Compute n = p × q and z = (p-1)(q-1)

Choose e with e < n such that gcd(e, z) = 1 (e and z share no common factors)

Find d such that e×d mod z = 1 (equivalently, ed ≡ 1 mod z). Find by trial or extended Euclidean algorithm.

Public key = (n, e). Private key = (n, d). Discard p, q (or keep secret).

Encryption: given message m (m must be < n): c = m^e mod n Decryption: given ciphertext c: m = c^d mod n Why it works: (m^e mod n)^d mod n = m^(ed) mod n = m^(ed mod z) mod n = m^1 mod n = m (Using the fact: a^y mod n = a^(y mod z) mod n, where n=pq, z=(p-1)(q-1)) Security: to find d from (n,e), you need to factor n into p and q. For 1024-bit primes, this takes longer than the age of the universe on current hardware.

RSA in Practice: Session Keys

RSA is computationally expensive (large number exponentiation). DES is ~100× faster than RSA. In practice: use RSA to establish a shared symmetric session key, then switch to symmetric encryption (e.g., AES) for the actual data. This gives you the key distribution advantage of public-key crypto with the speed of symmetric crypto.

Message Integrity: MACs and Digital Signatures

Message Authentication Code (MAC)

A MAC provides message integrity and authentication. Process: sender computes MAC = H(K_S || m) where H is a hash function (e.g., SHA-256) and K_S is the shared secret key. Receiver recomputes MAC and checks if it matches. If Trudy modifies m, MAC won't match. If Trudy doesn't know K_S, she can't forge a valid MAC.

MAC provides: integrity + authentication. It does NOT provide non-repudiation — since both Alice and Bob have K_S, either of them could have created the MAC. You can't prove to a third party which one sent it.

Digital Signature

A digital signature uses the sender's private key to sign: signature = K⁻(H(m)). Anyone with Alice's public key K⁺ can verify: K⁺(signature) = H(m) means Alice signed it (only Alice has K⁻).

Digital signature provides: integrity + authentication + non-repudiation. Because only Alice has her private key, only she could have produced the signature. Alice cannot later deny signing the message.

MAC
  • Uses shared symmetric key + hash
  • Fast (symmetric operations)
  • Provides: integrity, authentication
  • Does NOT provide: non-repudiation
  • Either party could have created it → can't prove to third party
Digital Signature
  • Uses private key to sign hash of message
  • Slower (asymmetric operations)
  • Provides: integrity, authentication, non-repudiation
  • Only signer has private key → verifiable by anyone with public key
  • Required for legally binding electronic contracts, etc.
★ Exam [Spring 2022 Q7b] — "MAC is not used in digital signature"

MAC requires both parties to know the same shared key. This means either party could have generated the MAC — there's no way to prove which one actually created the message. A judge or third party cannot verify who sent it. Digital signatures use the private key (known only to the signer) — anyone can verify the signature using the public key, proving it came from the private key holder. This is why digital signatures provide non-repudiation, while MAC does not — and hence MAC is insufficient for applications requiring proof of authorship (contracts, legal documents, etc.).
