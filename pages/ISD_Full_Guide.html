PART A — Information System Design (ISD)

ISD CH1

Information Systems & the Systems Analyst

Week 1

What is Information?

Information is an organizational resource that must be managed as carefully as physical resources like equipment or capital. Costs are associated with producing, storing, and retrieving information, so it must be managed efficiently to take full advantage of its potential. Raw data becomes information when it is processed and given context and meaning.

Eight Categories of Information Systems

TPS — Transaction ProcessingHandles daily routine transactions (sales, payroll). High volume. Example: ATM, POS terminals, online orders.
OAS — Office AutomationIncreases productivity of data workers. Word processors, spreadsheets, email, scheduling. Example: MS Office.
KWS — Knowledge WorkSupports knowledge workers (engineers, scientists). CAD tools, workstations. Creates new information/knowledge.
MIS — Management InfoProvides weekly/monthly summaries to middle management. Extracts from TPS. Structured decisions. Example: sales trend reports.
DSS — Decision SupportHelps managers make semi-structured decisions. "What-if" analysis. Interactive. Example: financial modeling tools.
ES — Expert SystemsCaptures expert knowledge as rules. AI-based. Mimics human expertise. Example: MYCIN (medical diagnosis).
GDSS — Group Decision SupportFacilitates group decision-making. Electronic meetings, voting, brainstorming tools. Reduces groupthink.
EES — Executive SupportFor senior executives. Aggregates internal + external data. Dashboard graphs. Exception reporting.

The Systems Analyst

The systems analyst is the key change agent who bridges users and technology. They act in three roles:

  • Outside consultant: Brings expertise to solve problems for organizations on contract
  • Supporting expert: Works within a company providing guidance to specific departments
  • Change agent: Plans and implements system changes, trains and motivates users

A good analyst must be: an effective communicator (oral and written), a problem solver, ethical with users and management, familiar with both technology and business processes, and able to manage time and people.

New Technologies Integrated into IS

  • E-commerce: Using the Web to perform business activities (buying, selling, customer service)
  • ERP (Enterprise Resource Planning): Integrates multiple IS across an organization (finance, HR, supply chain) into one system. Examples: SAP, Oracle ERP.
  • M-commerce: Wireless/handheld devices enabling transactions on the go
  • Open source software: Source code freely available to modify and distribute
ISD CH1

Systems Development Life Cycle (SDLC)

Week 1

Overview

The SDLC is a systematic 7-phase process for solving business problems through information systems. Errors found in later phases cost significantly more to fix than those caught in earlier phases — so thorough early work is critical.

The 7 Phases

PhaseKey ActivitiesPersonnel
1. Identify Problems, Opportunities & ObjectivesInterview managers; document problems; observe system; define scope and objectivesAnalyst, User management, Systems management
2. Determine Information RequirementsInterview users; use questionnaires; observe; gather documents. Learn the who/what/where/when/how/why.Analyst, User management, User operations workers
3. Analyze System NeedsCreate DFDs; complete data dictionary; make decisions; prepare system proposal; recommend solutionAnalyst, User management, Systems management
4. Design the Recommended SystemDesign UI, output, input, controls; design database; produce program specs; produce decision trees/tablesAnalyst, System designer, User management
5. Develop & Document SoftwareDesign using structure charts, N-S charts, pseudocode; code programs; write documentation; help filesAnalyst, System designer, Programmers
6. Test & Maintain SystemTest and debug programs; test system integration; enhance and fix errorsAnalyst, System designer, Programmers
7. Implement & Evaluate SystemPlan conversion; train users; purchase/install equipment; convert files; install system; review performanceAnalyst, System designer, Programmers, User management, User operations workers
Alternate Methodologies
  • Prototyping: Build a scaled-down version quickly to gather feedback before full development
  • RAD (Rapid Application Development): Object-oriented approach; emphasizes speed; uses CASE tools
  • ETHICS: Effective Technical and Human Implementation of Computer-based Systems — emphasizes human factors
  • Soft Systems Methodology: For complex, poorly-defined problems; focuses on different stakeholders' worldviews
  • Multi-view: Combines technical and social perspectives

CASE Tools

CASE (Computer-Aided Software Engineering) tools are automated software packages that assist in systems analysis and design. Benefits: increase analyst productivity, facilitate communication, provide continuity across life cycle phases, and assess maintenance impact.

Upper CASE (Front-end)
  • Used in analysis and design phases
  • Creates/modifies system design
  • Stores data in a project repository (records, diagrams, screens, reports)
  • Models organizational requirements
  • Defines system boundaries
Lower CASE (Back-end)
  • Generates computer source code from CASE design
  • Can generate in multiple languages
  • Reduces coding time and errors
  • Code is free from coding errors
  • Easier maintenance of generated code

Integrated CASE tools perform both upper and lower CASE functions in one package. Reverse Engineering is the opposite of code generation — it analyzes existing source code and converts it into CASE repository entities (data structures, screen designs, structure charts), producing documentation for poorly-documented legacy systems.

ISD CH3

Feasibility Study & Project Initiation

Week 1–2

Project Initiation

Projects are initiated for two broad reasons: problems that need system solutions, and opportunities for improvement. Organizational problems are identified by: checking output against performance criteria (errors, slow work), observing employee behavior (absenteeism, dissatisfaction, turnover), and listening to vendor/customer feedback (complaints, lost sales).

Five Criteria for Project Selection
  1. Backed by management (without support, projects fail)
  2. Timed appropriately for commitment of resources
  3. Moves the business toward its goals
  4. Practicable (doable within constraints)
  5. Important enough to be prioritized over other projects

Three Types of Feasibility

Technical FeasibilityCan the current technical resources support the new system? Can they be upgraded to the required level? Assesses hardware, software, network, and expertise availability.
Economic FeasibilityAre the time and money available to develop the system? Includes hardware, software, development costs vs. expected benefits. Needs cost-benefit analysis.
Operational FeasibilityAre human resources available to operate the system once installed? Will users accept it? Users who resist a system may make it operationally infeasible even if technically perfect.

Cost-Benefit Analysis

A tool to determine if the benefits of a proposed system outweigh its costs. Costs and benefits are either tangible (can be quantified in money) or intangible (real but difficult to quantify).

TypeCosts ExamplesBenefits Examples
TangibleHardware purchase, software licenses, development labor, training costsReduced staff costs, faster processing, fewer errors, increased sales
IntangibleLower employee morale during transition, organizational disruptionBetter customer service, improved decision making, competitive advantage
Cash Flow Analysis & Present Value

Because money has time value, costs and benefits occurring in different years must be converted to present value (PV) for a fair comparison.

Present Value (PV) = FV / (1 + i)^n Where: FV = Future Value (amount in future) i = interest rate (discount rate) per period n = number of periods in the future Example: A $10,000 benefit in 3 years at 10% interest rate: PV = 10,000 / (1.10)³ = 10,000 / 1.331 = $7,513 Net Present Value (NPV) = Sum of all PVs of benefits − Sum of all PVs of costs If NPV > 0: project is economically worthwhile If NPV < 0: costs outweigh benefits → reconsider
📌 Break-even Point
The year in which total accumulated benefits first exceed total accumulated costs. Plot cumulative costs and cumulative benefits on a graph; the crossing point is break-even.
ISD CH3

Project Scheduling & Management

Week 2

Activity Planning

Activity planning involves: selecting a systems analysis team, estimating time required for each task, and scheduling the project. Time estimates for each task can be made using three estimates:

Expected Time (ET) = (Optimistic + 4×Most_Likely + Pessimistic) / 6 (This is the weighted average used in PERT)

Gantt Charts

A Gantt chart is a horizontal bar chart that shows project activities (tasks) against time. Each bar represents one activity; its length shows the planned duration; overlap shows parallel activities. Easy to construct, read, and update.

  • Shows what activities are planned and when
  • Does NOT show dependencies between tasks (which tasks must finish before others begin)
  • Useful for: tracking progress, resource allocation, reporting to management

PERT Diagrams

PERT (Program Evaluation and Review Technique) diagrams show task precedence relationships — which activities must be completed before others can begin. Uses network diagram notation (nodes = events/milestones, arrows = activities).

  • Critical Path: The longest path through the network from start to finish. Determines the minimum time to complete the project. Any delay on the critical path delays the whole project.
  • Slack Time (Float): The amount of time a non-critical activity can be delayed without delaying the project. Slack = Late Finish − Early Finish.
  • Critical path activities have zero slack.
Critical Path Method (CPM): Forward pass: compute Early Start (ES) and Early Finish (EF) for each activity EF = ES + Duration ES of activity = max(EF of all predecessors) Backward pass: compute Late Start (LS) and Late Finish (LF) LF of last activity = EF of last activity (project deadline) LS = LF - Duration LF of activity = min(LS of all successors) Slack = LS - ES = LF - EF Critical activities: Slack = 0
Gantt Chart
  • Simple bar chart
  • Shows time and activities
  • Does NOT show dependencies
  • Easy for management to read
  • Good for progress tracking
PERT Diagram
  • Network diagram (nodes + arrows)
  • Shows task precedence/dependencies
  • Identifies critical path
  • Shows slack time on non-critical tasks
  • Better for complex projects
ISD INFO

Information Gathering — Interactive Methods

Week 3

Why Gather Information?

Phase 2 of the SDLC requires understanding the current system deeply. Information gathering answers: who uses the system, what information they need, where they get it from, when they need it, how it flows, and why things are done the way they are.

Interviews

Interviews are the most commonly used information gathering technique. They allow the analyst to probe for detail, observe non-verbal cues, and build rapport with users.

Steps in Interview Preparation

Research the interviewee and organization: Read annual reports, org charts, previous system documentation

Determine objectives of the interview: What specific information must you come away with?

Decide who to interview: Management for goals, operations staff for details and problems

Prepare questions: Mix open-ended and closed questions; sequence logically

Prepare the interviewee: Schedule in advance, explain purpose, build trust

Decide on recording method: Note-taking, audio recording (with permission), structured forms

Conduct the interview: Dress professionally, start with easy questions, listen actively, probe deeper with follow-ups

Write up results immediately after while memory is fresh. Send summary to interviewee for verification.

Types of Interview Questions
Open-EndedNo specific answer expected. "How do you feel about the current system?" Rich data, reveals attitudes. Time-consuming to analyze. Good for early discovery.
Closed / Closed-EndedLimited specific answers. "How many reports do you produce per week?" Quick, easy to analyze, comparable. Less depth. Good for facts and numbers.
BipolarSpecial case of closed. Yes/No or True/False. "Does the system produce daily reports?" Very restrictive. Use sparingly for confirmation only.
Probing / Follow-up"Can you give me an example?" or "Why is that important?" Follows up on an earlier answer to get depth. Essential for uncovering hidden requirements.
📌 Question Funnel
Best practice: Start an interview with broad open-ended questions (to discover issues you didn't know existed), then funnel down to specific closed questions (to get precise facts and confirm details). This is called the funnel approach. The opposite (inverted funnel) starts specific and broadens — useful when interviewee is shy or resistant.

Questionnaires

Questionnaires are useful when many people must be surveyed, when respondents are spread across locations, or when anonymity is important. They are cheaper per respondent than interviews but provide less depth.

Question Scales
Likert Scale5 or 7-point agreement scale. "Strongly Agree – Agree – Neutral – Disagree – Strongly Disagree." Measures attitudes and opinions. Easy to analyze statistically.
Semantic DifferentialBipolar adjective pairs with 7 positions. "Easy _:_:_:_:_:_:_ Difficult." Measures perceptions. Useful for UI evaluation.
Ranking ScaleRank items in order of preference (1 = best, etc.). Useful for prioritizing features. Forces comparison between options.
Rating ScaleRate each item independently (e.g., 1–10). Unlike ranking, multiple items can get the same score. Flexible and easy to use.
Questionnaire Design Guidelines
  • Use simple, clear language — avoid jargon
  • One topic per question — avoid double-barreled questions
  • Avoid leading or loaded questions
  • Include a mix of question types for variety
  • Test questionnaire on a small group before full distribution
  • Keep it short — response rate drops with length

Joint Application Design (JAD)

JAD is a structured group session that brings together users, management, and analysts to define system requirements collaboratively. Developed by IBM. Typically runs for 3–5 days in a workshop setting.

Benefits of JAD
  • All stakeholders in one room → faster decisions
  • Requirements agreed upon jointly → fewer changes later
  • Users feel ownership of the system
  • Reduces analysis and design time by up to 40%
  • Better quality requirements through group synergy
Drawbacks of JAD
  • Requires commitment of many busy people for days
  • Expensive (salaries + facilities for all participants)
  • Requires a skilled JAD facilitator to manage group dynamics
  • Dominant personalities can influence the group
  • May not work in highly political organizations
ISD INFO

Information Gathering — Unobtrusive Methods

Week 4

Why Unobtrusive Methods?

Unobtrusive (non-interactive) methods gather information without directly interviewing people. Useful when interviews are impractical, when you need objective data (people behave differently when watched), or to corroborate interview findings.

Sampling

Sampling is the process of systematically selecting representative elements from a large population for study, when examining every element is impractical or too expensive.

Why Sample?
  • The population is too large to study completely
  • Saves time and cost
  • A well-chosen sample gives accurate population estimates
  • Reduces analyst workload without sacrificing accuracy
Types of Sampling
Simple Random SamplingEvery element has equal probability of selection. Use random number tables. No bias. Best for homogeneous populations.
Systematic SamplingSelect every kth element (e.g., every 10th transaction). k = Population / Sample_size. Simple to implement. Can have periodic bias.
Stratified SamplingDivide population into subgroups (strata) by characteristic (e.g., department), then randomly sample from each. Ensures all groups are represented.
Convenience SamplingSelect whoever is available. Cheap and fast. Not representative — introduces bias. Use only for initial exploration.
Sample Size Considerations

Sample size depends on: population size, required confidence level (typically 95%), acceptable margin of error, and variability in the population. Larger, more variable populations require larger samples.

Observation

Directly watching how users interact with the current system. Provides insight into actual behavior (vs. what people say they do). Analysts observe workflows, bottlenecks, workarounds, and unofficial procedures.

  • Advantages: Reveals actual behavior, not reported behavior; finds undocumented procedures; builds understanding of work environment
  • Disadvantages: People behave differently when watched (Hawthorne Effect); time-consuming; may miss rare events; observer may misinterpret what they see

Analyzing Quantitative Documents

Existing documents reveal how the current system actually works and what data it processes. Types include: reports, forms, manuals, organizational charts, procedural documents, existing computer output.

Key questions when analyzing documents: What data is collected? How is it used? How often? Who produces it? Who uses it? What decisions depend on it? Are there gaps or redundancies?

ISD DFD

Data Flow Diagrams (DFDs)

Week 5

What are DFDs?

A Data Flow Diagram (DFD) graphically represents how data moves through an information system — showing processes, data stores, external entities, and the flows of data between them. DFDs model the logical system (what it does) not the physical system (how it does it).

Four DFD Components

SymbolNameDescriptionYourdon NotationGane & Sarson
External EntitySources & SinksA person, organization, or system outside the system boundary that sends or receives dataRectangle / SquareRectangle
ProcessTransformationTransforms input data into output data. Must have at least one input and one output flowCircle (bubble)Rounded rectangle
Data StoreStorageRepository of data at rest — files, databases, folders. Data is referenced or updated by processesTwo parallel lines (open rectangle)Rounded rectangle (closed)
Data FlowMovementMovement of data between any two DFD elements. Labeled with the data name. Shown as directed arrow.Labeled arrowLabeled arrow
📌 DFD Rules
  • Processes must have at least one input AND one output flow
  • Data cannot flow directly between two external entities
  • Data cannot flow directly between two data stores
  • External entities cannot connect to data stores directly (must go through a process)
  • All data flows must be named (except flows to/from data stores that are obvious)

DFD Levels

Context Diagram (Level 0)The highest-level view. Shows the ENTIRE system as a SINGLE process bubble. Shows all external entities and the data flows in/out of the system boundary. No data stores. Purpose: define scope.
Level 1 (System Diagram)Explodes the single context bubble into the major subsystems (3–7 processes). Shows primary data flows and data stores between these major processes. First level of detail.
Level 2Further explodes each Level 1 process into sub-processes. More detail. Continue until each process is simple enough to describe in 1 paragraph (primitive DFD).
✓ How to Draw a DFD from Requirements
  1. Identify all external entities (who interacts with the system from outside)
  2. Identify all data flows going in/out of the system boundary
  3. Draw the context diagram with one process bubble
  4. Break down the single process into major subsystem processes (Level 1)
  5. Add data stores between processes where data is saved
  6. Draw data flows between processes and stores, labeling each
  7. Check DFD rules — balance and consistency between levels

Data Dictionary

The data dictionary is a structured repository that defines all data elements referenced in DFDs — every flow, store, process, and external entity is formally defined. It ensures consistency across the system design.

Data dictionary notation: = is composed of (definition) + and (concatenation) {} iteration (one or more occurrences) () optional element [] selection (one of the listed options) * comment Example: Customer_Order = Customer_ID + Customer_Name + {Order_Line} + Order_Date Order_Line = Product_Code + Quantity + Unit_Price
ISD UML

Object-Oriented Analysis & UML

Week 6

Object-Oriented Concepts

ObjectAn instance of a class — a real-world thing with attributes (data) and methods (behaviors). Example: Customer object with name, address attributes and placeOrder() method.
ClassA template/blueprint for creating objects. Defines attributes and methods shared by all instances. Written as: ClassName / attributes / methods (3-part box).
InheritanceA subclass inherits all attributes and methods of its superclass (parent). Allows reuse. Example: Car inherits from Vehicle (has wheels, engine) and adds specific attributes.
EncapsulationBundling data (attributes) and methods within a class and hiding internal details from outside. Only the interface (public methods) is exposed.
PolymorphismSame method name behaves differently in different classes. Example: draw() method on Circle draws a circle; on Rectangle draws a rectangle.
AssociationA relationship between classes. Types: Association (uses), Aggregation (has-a, weak), Composition (contains, strong), Dependency (depends on).

CRC Cards (Class-Responsibility-Collaborator)

CRC cards are index cards used to model classes during early OO analysis. Each card represents one class and has three sections: the Class name at the top, Responsibilities (what it knows/does) on the left, and Collaborators (which other classes it needs help from) on the right.

They are used in brainstorming sessions — physical cards can be moved around the table, enabling teams to visualize object interactions without committing to a formal diagram. XP (Extreme Programming) actively promotes their use.

UML Diagrams

Use Case Diagram

Shows the functional requirements of a system from the user's perspective. Components:

  • Actor: Stick figure — represents a user or external system that interacts with the system
  • Use Case: Oval — represents a specific function or goal the system provides to actors
  • System Boundary: Rectangle — defines what is inside vs outside the system
  • Association: Line between actor and use case — shows participation
  • <<include>>: One use case always includes behavior of another (mandatory reuse)
  • <<extend>>: One use case optionally extends another (conditional behavior)
  • Generalization: Inheritance between actors or use cases
Class Diagram

The most important UML diagram — shows the static structure of the system. Each class is a 3-compartment box: class name / attributes (with types) / methods (with signatures). Shows relationships between classes with labeled, directed arrows and multiplicity (1..*, 0..1, etc.).

Relationship types in class diagrams: Association: ───── "A uses B" (general) Aggregation: ────◇ "A has B" (weak has-a; B can exist without A) Composition: ────◆ "A contains B" (strong has-a; B cannot exist without A) Inheritance: ────▷ "A is a B" (generalization; open triangle arrowhead) Dependency: - - → "A depends on B" (dashed arrow) Multiplicity examples: 1 exactly one 0..* zero or more 1..* one or more 0..1 zero or one
Sequence Diagram

Shows the time-ordered interactions between objects for a specific scenario/use case. Components:

  • Objects: Boxes at the top with dashed vertical lifelines below them
  • Activation bar: Thin rectangle on lifeline — when object is active/executing
  • Messages: Horizontal arrows between lifelines — show method calls (solid arrow) and returns (dashed arrow)
  • Time flows downward
Activity Diagram

Shows the workflow / business process flow — similar to a flowchart but for OO systems. Shows the sequence of activities, decision points (diamonds), parallel actions (fork/join bars), and start/end states (filled circles). Good for modeling business processes and use case flow of events.

Use Case Diagram
  • Shows WHAT the system does
  • Functional requirements
  • User perspective
  • No internal implementation shown
Sequence Diagram
  • Shows HOW objects interact
  • Temporal order of messages
  • One scenario at a time
  • Shows internal collaboration
ISD TEST

Testing Conventional Applications

Week 7

Testing Levels

Unit TestingTests individual program modules/functions in isolation. Finds coding errors. Performed by developers during development (Phase 5).
System TestingTests the fully integrated system as a whole against the original system requirements. Does the whole thing work?
Alpha TestingPerformed by internal users (inside the developing organization) in a controlled environment. Simulated or actual operational environment.
Beta TestingPerformed by external users (real customers) in their own environment with real data. Final validation before release.
Acceptance TestingFormal testing conducted by the client/end user to determine if the system meets their requirements before accepting delivery.
DebuggingThe process of finding and fixing errors after a test has revealed them. Testing reveals THAT there is an error; debugging finds WHERE it is and fixes it.

White Box Testing (Structural Testing)

White box (glass box) testing examines the internal logic and structure of the program. The tester has full knowledge of the source code. Tests are designed to ensure every line of code, every branch, and every path is executed at least once.

Basis Path Testing & Cyclomatic Complexity

Cyclomatic complexity (V(G)) measures the structural complexity of a program and gives the number of linearly independent paths — the minimum number of test cases needed for complete path coverage.

Three equivalent formulas for Cyclomatic Complexity V(G): Method 1 (Edges & Nodes): V(G) = E − N + 2P where: E = number of edges (arrows) in the control flow graph N = number of nodes (boxes) in the control flow graph P = number of connected components (usually P=1 for one program) Method 2 (Regions): V(G) = number of enclosed regions + 1 (the outer region) (Count the areas enclosed by the graph, add 1 for the unbounded outer area) Method 3 (Predicates): V(G) = number of predicate (decision) nodes + 1 (Count every IF, WHILE, FOR, CASE, AND, OR — each is a predicate node) Example: A program with IF (2 branches) and a WHILE loop Predicate nodes = 2 (the IF + the WHILE) → V(G) = 2 + 1 = 3 Must write at least 3 test cases to cover all independent paths

Draw the control flow graph from the code (each statement = node, decision = branch)

Calculate V(G) using any of the three methods above

The result = number of independent paths = minimum number of test cases

Identify each independent basis path through the graph

Design test cases that force execution of each path

Black Box Testing (Functional Testing)

Black box testing treats the program as a black box — the tester has no knowledge of the internal code. Tests are based purely on the specification: given this input, is the output correct? Focuses on functionality, not structure.

Equivalence Partitioning

Divide the input domain into classes of equivalent inputs — any value from one class should behave the same as any other value in that class. Choose one representative value from each partition as a test case, reducing the total number of needed tests.

Example: A field accepts integers 1–100 Valid partitions: {1 to 100} → test with e.g. 50 Invalid partitions: {less than 1} → test with e.g. -5 or 0 {greater than 100} → test with e.g. 150 {non-integers} → test with e.g. "abc"
Boundary Value Analysis (BVA)

Errors tend to cluster at the edges of input domains. Test at the exact boundary AND just inside/outside it.

Example: Valid input range is 1 to 100 Test cases: 0 (just below), 1 (lower boundary), 2 (just above lower), 50 (typical middle), 99 (just below upper), 100 (upper boundary), 101 (just above) (7 boundary test cases for one range)

Gray Box Testing

Gray box testing combines elements of both white and black box testing. The tester has partial knowledge of the internal workings (e.g., database schema, architecture, algorithms) but tests the system externally like a black box. Useful for integration testing between modules developed by different teams.

FeatureWhite BoxGray BoxBlack Box
Knowledge of codeFullPartialNone

SE CH1

Software & Software Engineering Introduction

Week 8

What is Software?

Software is: (1) instructions (programs) that when executed produce desired features, function, and performance; (2) data structures that enable programs to manipulate information; (3) documentation that describes operation and use of the programs.

Software Characteristics
  • Engineered, not manufactured: Software is designed and built by human logic — not produced on an assembly line
  • Doesn't wear out: Software has no physical degradation. But it deteriorates due to change — each fix may introduce new bugs, increasing failure rate over time
  • Component-based construction: Industry is moving this way, but most software is still custom-built
Wear vs Deterioration
  • Hardware: failure rate is high at start (defects), drops to steady low (normal operation), then rises again as physical components wear out (bathtub curve)
  • Software: failure rate is high initially (defects), drops after fixing, but each change can introduce new failures → deterioration over time, not wear
  • This is why software maintenance is so costly (48–60% of total time)

Software Application Categories

System SoftwareServes other programs. OS, compilers, device drivers, network software. Extensive data structures, complex algorithms. Must be efficient.
Application SoftwareSolves specific business needs. Payroll, ERP, CRM. Processes business or technical data. Largest category.
Engineering/ScientificNumber crunching algorithms. Astronomy, engineering, molecular biology simulations. CAD/CAM, weather modeling.
Embedded SoftwareLives in ROM inside hardware. Fuel injection control, elevator systems, keypad on microwave. Very constrained environment.
Product-Line SoftwareDelivers specific functionality to many customers. Word processors, spreadsheets. Addresses marketplace of general users.
WebAppsNetwork-intensive, concurrent, evolving continuously. Presents text, graphics, audio/video. Must be available 24/7/365.
AI SoftwareUses non-numerical algorithms. Expert systems, pattern recognition, neural networks, natural language processing.

Legacy Software

Legacy software = old systems that are difficult to modify but still critical to business operations. It must be changed because: it must adapt to new computing environments; business requirements change; it must interoperate with modern systems; it must be re-architected for network environments. Simply replacing legacy systems is often too risky and expensive.

Software Engineering — Definition & Layered Technology

IEEE definition: "The application of a systematic, disciplined, quantifiable approach to the development, operation, and maintenance of software."

Software engineering is organized as a layered technology:

Tools ← automate methods (CASE tools, IDEs, debuggers) Methods ← technical how-to's (analysis, design, coding, testing) Process Model ← framework gluing methods together (Waterfall, Agile, etc.) Quality Focus ← foundation — commitment to quality underlies everything

Process Framework

Every software process model includes a set of framework activities that apply regardless of which model is used:

CommunicationUnderstand stakeholder objectives and requirements before work begins
PlanningEstimate effort, schedule, risk; define tasks and deliverables
ModelingAnalysis of requirements; design of architecture, components, interfaces
ConstructionCode generation + testing at unit and integration levels
DeploymentDeliver to customer; evaluate; receive feedback for next increment

Umbrella Activities run across all framework activities: project management, formal technical reviews, SQA (Software Quality Assurance), configuration management, work product preparation, reusability management, measurement, risk management.

SE CH4

Software Process Models

Week 8–9

1. Waterfall Model

The classic sequential process. Each phase must complete fully before the next begins. No going back.

Requirements → System Design → Implementation (coding) → Testing → Deployment → Maintenance
When to Use
  • Requirements are very well documented, clear, and fixed
  • Technology is well understood and stable
  • No ambiguous requirements
  • Short project with stable team
  • Similar projects done before
Disadvantages
  • Doesn't allow for reflection or revision mid-project
  • Real projects rarely have fully stable requirements
  • Customer only sees the system at end (too late for changes)
  • Working software not produced until late in lifecycle
  • High risk for complex, novel systems

2. V-Model

An extension of Waterfall that explicitly shows the relationship between each development stage and its corresponding test phase. The left side descends through development; the right side ascends through testing. Each development phase has a corresponding validation/verification test phase.

Requirements Analysis ←——→ Acceptance Testing System Design ←——→ System Testing Architecture Design ←——→ Integration Testing Module Design ←——→ Unit Testing Implementation (Coding) at the bottom

Advantage over Waterfall: testing is planned early (not afterthought). Verification at each level prevents defects from propagating down.

3. Incremental Model

Delivers the system in a series of increments. Each increment adds more functionality. The first increment is often the core product. Users can use partial functionality early and provide feedback for subsequent increments.

  • Each increment is a mini-waterfall (requirements → design → code → test)
  • Earlier increments serve as prototypes for refining later ones
  • Easier to manage risk — if budget runs out, you still have a working (partial) system

4. RAD (Rapid Application Development) Model

High-speed adaptation of Waterfall using component-based construction. Very short development cycle (60–90 days). Works only when requirements are well understood and scope is constrained. Multiple teams work in parallel on different components.

RAD Phases: Business Modeling → Data Modeling → Process Modeling → Application Generation (reuse components) → Testing and Turnover

Limitation: Requires many developers for parallel work; needs modular architecture; doesn't work well when technical risks are high or system can't be modularized.

5. Evolutionary Models

a. Prototyping Model

Build a quick, scaled-down version of the system to help users and developers understand requirements. The prototype is iteratively refined based on feedback. Eventually the prototype becomes (or is discarded and replaced by) the real system.

Quick Plan → Quick Design → Build Prototype → Customer Evaluation → Refine Prototype → repeat

Problem: Customers may think the prototype IS the final system. Developers may make permanent bad technical choices just to get the prototype working quickly.

b. Spiral Model

Combines the iterative nature of prototyping with the systematic aspects of waterfall. Each cycle through the spiral represents a phase of the project. Most importantly — it explicitly includes risk analysis in every cycle.

Each spiral cycle has 4 quadrants: 1. Determine objectives, alternatives, and constraints 2. Evaluate alternatives, identify and resolve RISKS (risk analysis) 3. Develop and verify next-level product 4. Plan the next phase (review and commit)

The radius of the spiral represents the cumulative cost; each pass adds cost and completes more of the product. Best for large, high-risk projects where risk management is critical.

c. Concurrent Development Model

In reality, multiple software engineering activities happen simultaneously (concurrently) in different states. A requirements activity may be "awaiting changes" while design is "under development" while coding is "under review." All activities shown with their current states at any time.

6. Unified Process (UP)

A use-case driven, architecture-centric, iterative and incremental process framework based on UML. Organizes work by phases (time-based) crossed with workflows (activity-based).

PhasePurposeKey Milestone
InceptionScope, feasibility, high-level requirements, initial use casesLifecycle Objectives
ElaborationDetailed requirements, architecture, risk identification and resolutionLifecycle Architecture
ConstructionBuild the system incrementally; all use cases implemented and testedInitial Operational Capability
TransitionDeploy to users; beta testing; training; final releaseProduct Release

Activities like requirements, design, coding, and testing all occur throughout all phases — but their relative intensity changes. Requirements-heavy in Inception; design-heavy in Elaboration; construction-heavy in Construction phase.

Agile Development, XP & Scrum

Week 9, 12 (partial)

What is Agility?

Agility = effective (rapid and adaptive) response to change. Agile processes recognize that plans are short-lived, that requirements change frequently, and that close customer collaboration is essential. The cost of change is kept relatively flat throughout the project (unlike waterfall where changes become exponentially expensive late in the project).

Agile Manifesto — 4 Core Values
Value (left) over (right)
  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan
Agile Process Characteristics
  • Driven by customer descriptions (user stories)
  • Recognizes plans are short-lived
  • Iterative development with heavy construction emphasis
  • Delivers multiple software increments
  • Adapts as changes occur

Extreme Programming (XP)

The most widely used agile process, proposed by Kent Beck. XP pushes engineering practices to an extreme.

XP PlanningBegins with user stories (index cards describing features). Team assigns cost/effort to each. Stories grouped into deliverable increments. Commitment made on delivery date. Project velocity (from 1st increment) guides future estimates.
XP DesignKISS principle (Keep It Simple, Stupid). Uses CRC cards. Creates "spike solutions" for hard design problems (quick prototype). Encourages refactoring — iterative improvement of internal design without changing behavior.
XP CodingUnit tests written BEFORE code (TDD — Test-Driven Development). Pair programming: two developers share one keyboard — one codes, one reviews in real time. Continuous code integration.
XP TestingAll unit tests executed daily. Acceptance tests defined by customer to assess customer-visible functionality. Continuous automated testing. If a test fails, it must be fixed immediately.

Scrum

A lightweight agile framework focused on managing and controlling iterative development. Organizes work into fixed-length Sprints (typically 2–4 weeks).

Product BacklogPrioritized list of all features/requirements (user stories) for the entire product. Owned and prioritized by Product Owner. Living document — always changing.
SprintFixed-length iteration (2–4 weeks). Team selects items from backlog for this sprint → Sprint Backlog. No changes allowed to sprint scope once started.
Daily Scrum / Standup15-minute daily meeting. Each member answers: What did I do yesterday? What will I do today? What blocks me?
Sprint ReviewTeam demonstrates working software to stakeholders at sprint end. Inspect and adapt.
Product OwnerRepresents customer. Prioritizes Product Backlog. Defines acceptance criteria.
Scrum MasterFacilitates Scrum process. Removes impediments. Not a manager — a servant-leader.
SE CH10

Software Project Estimation

Week 10–11

Project Planning Overview

Software project planning produces a plan that provides a road map for the software team. It begins with scope definition — understanding what the software is supposed to do and what its bounds are.

Resource Categories
  • Human resources: People with appropriate skills (number + position + specialty)
  • Software resources: Reusable components, tools, CASE environment, development software
  • Hardware/Platform resources: Target hardware, development hardware, network environment

LOC-Based Estimation (Lines of Code)

Estimate the size of the software in lines of code (LOC), then use historical productivity data to estimate effort.

Step 1: Estimate LOC for each system function/component Use 3-point estimate: LOC_expected = (LOC_optimistic + 4×LOC_most_likely + LOC_pessimistic) / 6 Step 2: Compute total estimated LOC = sum of all component LOCs Step 3: Apply productivity rate from historical data Productivity = LOC / person-month (from past similar projects) Effort (person-months) = Total_LOC / Productivity Step 4: Estimate duration Duration = Effort^(1/3) × constant (Putnam model, approximately) OR Duration = Effort / Team_size Step 5: Estimate cost Cost = Effort (person-months) × Labor_rate ($/person-month) Example: Estimated 12,000 LOC; Productivity = 600 LOC/person-month Effort = 12,000 / 600 = 20 person-months Duration ≈ 20 / 4-person team = 5 months Cost = 20 × $5,000/month = $100,000
📌 LOC Limitations
LOC is language-dependent (1000 lines of assembly ≠ 1000 lines of Python). It counts what was written, not what was accomplished. Hard to estimate accurately before coding. Doesn't account for reuse.

Function Point (FP) Based Estimation

Function Points measure software size by the amount of functionality delivered to the user, independent of programming language. Developed by Allan Albrecht at IBM.

VAF = 0.65 + 0.01 × Σ(Fi) where i = 1 to 14 The 14 General System Characteristics (Fi), each rated 0–5: 0 = No influence, 1 = Incidental, 2 = Moderate, 3 = Average, 4 = Significant, 5 = Essential 1. Data communications 2. Distributed data processing 3. Performance requirements 4. Heavily used configuration 5. Transaction rate 6. Online data entry 7. End-user efficiency 8. Online update 9. Facilitate change Range of VAF: Min: 0.65 + 0.01×0 = 0.65 Max: 0.65 + 0.01×70 = 1.35 (VAF can adjust UFP by at most ±35%)
Step 4: Calculate Adjusted Function Points (FP)
FP = UFP × VAF Then use FP for estimation: Effort (person-months) = FP / Productivity_Rate (FP/person-month from history) Cost = Effort × Labor_Rate LOC_estimate = FP × LOC_per_FP (language-dependent conversion factor) (e.g., Java: ~55 LOC/FP; C: ~130 LOC/FP; COBOL: ~106 LOC/FP)
