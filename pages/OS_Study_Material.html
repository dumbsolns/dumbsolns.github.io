OS INTRO

Introduction to Operating Systems

OS Architectures

Monolithic Kernel
  • Entire OS in one large binary in kernel mode
  • All services (FS, drivers, network) run in kernel space
  • Pros: Fast — no IPC overhead between components
  • Cons: Huge size; any crash kills entire OS; recompile whole kernel for any change; poor maintainability
  • Example: early UNIX
Modular Kernel (Modern)
  • Core kernel in RAM; other services loaded as modules
  • Modules loaded/unloaded dynamically without reboot
  • Pros: Easy maintenance — change one module without recompiling everything; strip down for embedded use
  • Used by: Linux, Windows, Android
Microkernel
  • Kernel has only: basic scheduling, IPC, low-level memory
  • File system, drivers, network run in user space as servers
  • Communication via message passing (IPC)
  • Pros: Crash in user-space server doesn't crash kernel; more secure; easier to extend
  • Cons: Slower — every service call needs IPC (mode switches)
  • Examples: Mach, QNX, L4
Hypervisors
  • Type 1 (Bare Metal): runs directly on hardware, no host OS. Higher performance, security. Used in data centers. Examples: ESXi, Hyper-V, KVM
  • Type 2 (Hosted): runs on top of a host OS like an application. Easier to install, lower performance. Examples: VirtualBox, VMware Workstation, Parallels
  • Differentiate Monolithic vs Modular vs Microkernel with figures — appears in EVERY paper
  • Microkernel key: user-space servers + IPC. Modular key: loadable kernel modules, no IPC overhead
  • Type 1 vs Type 2 Hypervisor with examples [almost every exam since 2018]
  • VDI (Virtual Desktop Infrastructure): centralized server hosts desktop VMs; accessed remotely; Persistent VDI = retains user data; Non-Persistent = resets on logout

System Calls and Kernel Mode

System calls are the interface between user programs and the OS. When a process needs OS services (read file, allocate memory), it triggers a mode switch from user mode → kernel mode. The OS uses a dispatch table (Linux: sys_call_table; Windows: SSDT) indexed by system call number to find the right kernel function.

The dispatcher handles context switches between processes. Dispatch latency = time for the dispatcher to stop one process and start another. Must be minimized since it runs on every context switch.

CPU SCHED

CPU Scheduling

Scheduling Criteria and Key Formulas

Turnaround Time (TT) = Finish Time (FT) - Arrival Time (AT) Waiting Time (WT) = TT - CPU Burst Time = FT - AT - burst Response Time (RT) = First Start Time - AT (first time CPU given to process) Average TT / WT / RT = sum / number of processes

Scheduling Algorithms

AlgorithmTypeRuleKey Problem
FCFSNon-preemptiveFirst arrived → first servedConvoy effect — short jobs stuck behind long ones
SJFNon-preemptiveShortest burst next; optimal avg waitStarvation of long jobs; needs burst time in advance
SRTNPreemptiveIf new job has shorter remaining time → preemptSame starvation as SJF

Round RobinPreemptiveEach process runs for time quantum q, then preemptedHigh overhead if q too small; FCFS if q → ∞
MLFQPreemptiveMultiple queues; CPU-bound demoted; I/O-bound stays highComplex; starvation of lower queues without aging

Round Robin — Step-by-Step

At time 0, add all processes with AT=0 to queue (alphabetical order for ties)
Run first process for min(q, remaining_burst). When done, remove from queue
After running, add any NEW processes that arrived during that quantum to the queue (before the preempted process rejoins)
Preempted process goes to back of queue. Repeat until all done
For each process: RT = first start − AT; FT = time it finishes; TT = FT − AT; WT = TT − burst
  • RR appears in EVERY exam — must draw full Gantt chart, track remaining bursts, compute all 3 metrics
  • Tie-breaking rule: newly arrived processes join queue BEFORE preempted process when both happen at same time
  • If process finishes before quantum ends, next process starts immediately (no wasted time)

Multilevel Feedback Queue (MLFQ)

Multiple queues, each with a different time quantum. Process starts in Q1 (highest priority, smallest quantum). If it uses full quantum without finishing → demoted to Q2 (larger quantum). Continue until lowest queue. I/O-bound processes complete before quantum → stay in high queues.

  • Typical MLFQ exam setup: Q1 (q=7 or 8), Q2 (q=12–18), Q3 (q=20–25), Q4 = FCFS or FCFS/SJF [recurring pattern in Fall 2021–2024]
  • Track which queue each process is currently in. Process demotes if uses full quantum.
  • If process doesn't use full quantum (I/O bound or finishes) → stays in same queue or leaves
  • Preemption: higher queue always preempts lower queue (a process arriving in Q1 preempts a Q2 process)
  • Draw the Gantt chart step by step, tracking queue levels. Compute WT, RT, TT same way.

Weighted Round Robin

Weight_i = Priority_i × Weight_Multiplier Total_Weight = sum of all process weights currently in system vruntime_i = floor( (Weight_i / Total_Weight) × Scaling_Value ) (minimum vruntime = 1; floor = round down) Each round: every ready process runs for exactly its vruntime. New arrivals: join at start of NEXT round (not mid-round).
  • Step 1: compute weights. Step 2: sum weights of processes active in that round. Step 3: compute vruntime for each. Step 4: draw Gantt (each process runs its vruntime in round-robin order). Step 5: compute metrics.
  • When a process arrives mid-round, it joins at the NEXT round. Recalculate total weight and vruntimes for that new round.
  • If a process finishes mid-round (remaining burst < vruntime), it runs only remaining burst, then exits. Adjust schedule accordingly.

CPU Burst Time Estimation

τ_{n+1} = α × t_n + (1 - α) × τ_n [α typically = 0.5] Where t_n = actual burst, τ_n = predicted burst, α = weighting factor Each past sample's influence decays exponentially. With α=0.5: τ = (t_n + t_{n-1} + t_{n-2}/2 + ...) / running sum

Multiprocessor Scheduling

SMP (Symmetric)
  • All CPUs are equal peers, each runs own scheduler
  • Modern standard: Linux, Windows
  • Needs load balancing: push migration (overloaded CPU pushes tasks) or pull migration (idle CPU pulls tasks)
  • Processor affinity: keep process on same CPU (warm cache). Soft = preferred; Hard = enforced
AMP (Asymmetric)
  • One master CPU runs OS kernel and scheduler
  • Other CPUs run user processes only
  • Master becomes bottleneck as system scales
  • Simpler to implement but less scalable
DEAD

Deadlocks

Four Necessary Conditions (Coffman Conditions)

ALL four must hold simultaneously for deadlock to occur. Prevent any one → prevent deadlock.

1. Mutual ExclusionResource can only be used by one process at a time. Can't negate for inherently non-sharable resources.
2. Hold and WaitProcess holds ≥1 resource while waiting for more. Prevent: require all resources upfront, or release all before requesting new ones.
3. No PreemptionResources released only voluntarily. Prevent: force release if process must wait (only works for preemptible resources like CPU/memory).
4. Circular WaitA cycle of processes, each waiting for a resource held by the next. Prevent: impose total ordering on resources — always request in increasing order.
  • RAG cycle detection: DFS from each node. If you revisit a node already in current path → cycle → possible deadlock
  • Single-instance resources: cycle = definite deadlock. Multi-instance: cycle = possible deadlock (not guaranteed)
  • Wait-for graph: remove resource nodes from RAG; edge Pi→Pj means Pi waits for a resource held by Pj

Banker's Algorithm (Deadlock Avoidance)

Data structures (n processes, m resource types): Available[m] = total − sum(Allocation) for each resource Max[n][m] = max demand declared by each process Allocation[n][m] = currently allocated Need[n][m] = Max − Allocation ← compute this first! Safety Algorithm: Work = Available.copy() Finish[] = {false, false, ..., false} Loop: Find i: Finish[i]=false AND Need[i] ≤ Work If found: Work += Allocation[i]; Finish[i] = true; repeat If not found: break All Finish[i]=true → SAFE STATE → safe sequence found Any Finish[i]=false → UNSAFE STATE → possible deadlock Resource Request for process Pi (Request vector): 1. If Request > Need[i] → ERROR (exceeded max claim) 2. If Request > Available → WAIT 3. Hypothetically allocate: Available -= Request Allocation[i] += Request Need[i] -= Request 4. Run Safety Algorithm 5. If SAFE → grant. If UNSAFE → rollback and make Pi wait.
  • The most common exam question — always compute Need first: Need = Max − Allocation
  • Available = Total Resources − column sums of Allocation
  • Safe sequence: at each step, find a process whose Need ≤ current Work, add its allocation to Work
  • Can multiple safe sequences exist? YES — any valid ordering works
  • For resource requests: check Request ≤ Need (else error) then Request ≤ Available (else wait) then hypothetical allocation → safety check
  • P4 requests (3,3,0): check (3,3,0) ≤ Need[P4] and (3,3,0) ≤ Available. If both yes, hypothetically allocate and run safety. If still safe, grant.

Full Worked Example (Classic 5-Process, 3-Resource)

Resources: A=10, B=5, C=7 Allocation Max Need Available A B C A B C A B C A B C P0: 0 1 0 7 5 3 7 4 3 → 3 3 2 P1: 2 0 0 3 2 2 1 2 2 P2: 3 0 2 9 0 2 6 0 0 P3: 2 1 1 2 2 2 0 1 1 P4: 0 0 2 4 3 3 4 3 1 Safety check (Work starts at 3,3,2): P1: Need(1,2,2) ≤ (3,3,2)? ✓ → Work=(5,3,2) P3: Need(0,1,1) ≤ (5,3,2)? ✓ → Work=(7,4,3) P4: Need(4,3,1) ≤ (7,4,3)? ✓ → Work=(7,4,5) P2: Need(6,0,0) ≤ (7,4,5)? ✓ → Work=(10,4,7) P0: Need(7,4,3) ≤ (10,4,7)? ✓ → Work=(10,5,7) Safe sequence: <P1, P3, P4, P2, P0>

Deadlock Recovery

  • Resource Preemption: Take resource from one process, give to another. Criteria: minimize cost, choose low-priority process, process that can be rolled back
  • Process Termination: Kill all deadlocked (disruptive) OR kill one at a time and recheck (less disruptive). Choose victim by: priority, resources held, progress made, rollback feasibility
  • Rollback: Return process to safe checkpoint taken earlier, retry. Starvation risk: same process always picked → include rollback count in cost
SYNC

Process Synchronization

Race Conditions and Critical Sections

A race condition: two or more processes access shared data concurrently and the final result depends on execution order. A critical section (CS) is the code segment accessing shared data. Requirements: Mutual Exclusion (only one in CS), Progress (no unnecessary blocking), Bounded Waiting (no starvation).

Why plain lock variable fails: if(lock==0){ lock=1; ... } — the read and write are NOT atomic. Two processes can both read lock=0 before either writes lock=1 → both enter CS. Need hardware-level atomicity.

Priority Inversion: High-priority H busy-waits for Low-priority L to release a lock. But H's running prevents L from getting CPU → L never releases → H loops forever. Solution: Priority Inheritance (raise L to H's priority temporarily).

Semaphores

Semaphore S = integer variable. Only two atomic operations: down(S) / wait(S) / P(S): up(S) / signal(S) / V(S): if S > 0: S--; proceed S++ else: BLOCK (sleep) if(any blocked): wake one up Binary semaphore (mutex): S ∈ {0,1}. Initialize to 1. Protects critical section. Counting semaphore: S = N (number of resources). Controls access to N identical resources.

Producer-Consumer (Bounded Buffer)

mutex = 1 // mutual exclusion for buffer empty = N // empty slots (initially N) full = 0 // full slots (initially 0) Producer: Consumer: while(TRUE){ while(TRUE){ produce_item(); down(&full); // wait for item down(&empty); // need space down(&mutex); // lock buffer down(&mutex); // lock item = remove(); add_item(); up(&mutex); up(&mutex); up(&empty); // free a slot up(&full); // item added consume(item); } }
  • Always 3 semaphores: mutex (init=1), empty (init=N), full (init=0)
  • Producer: down(empty) THEN down(mutex) — order matters to avoid deadlock
  • Consumer: down(full) THEN down(mutex)
  • Scenario: buffer full → producer blocks on down(empty) until consumer consumes and calls up(empty)

Readers-Writers

mutex = 1 // protects read_count db = 1 // exclusive DB access read_count = 0 // number of active readers Reader: Writer: down(mutex); read_count++; down(db); // exclusive access if(read_count==1) down(db); write_data(); up(mutex); up(db); read_data(); down(mutex); read_count--; if(read_count==0) up(db); up(mutex);

First reader locks writers out (down(db)). Last reader releases (up(db)). Any number of readers simultaneously — only writers need exclusive access. Writers can starve with basic solution (fair version: when writer waiting, no new readers).

Sleeping Barber

customers=0, barbers=0, mutex=1, waiting=0, CHAIRS=N Barber: Customer: while(TRUE){ down(mutex); down(&customers); // sleep if 0 if(waiting < CHAIRS){ down(&mutex); waiting++; waiting--; up(&customers); // wake barber up(&barbers); // barber ready up(&mutex); up(&mutex); down(&barbers); // wait for barber cut_hair(); get_haircut(); } } else up(&mutex); // shop full, leave
  • Modified versions: 3 barbers + 6 chairs → barbers = 3 initially (not 0); CHAIRS = 6
  • Justify scenarios: (1) barber enters empty shop → sleeps (blocks on customers semaphore) (2) customer arrives → wakes barber via up(customers) (3) all chairs full → customer leaves without waiting (4) multiple barbers: barbers semaphore tracks available barbers
PROC

Processes and Threads

Processes

A process = program in execution. Contains: text (code), data (globals), heap (dynamic), stack (function frames, local vars), program counter, CPU registers. Process state: New → Ready → Running ↔ Waiting → Terminated.

PCB (Process Control Block) stores: state, PC, registers, scheduling info (priority, queue pointers), memory info (page tables), accounting (CPU time used), I/O status (open files, devices).

CPU Utilization = 1 - p^n (p = fraction of time processes wait for I/O; n = processes in memory) Example: p=0.8, n=10: utilization = 1 - 0.8^10 = 1 - 0.107 = 89.3% Need ~10 processes in memory to keep CPU 90% busy when I/O wait = 80%

Threads

A thread is the smallest execution unit within a process. Multiple threads in a process share: code, data, heap, open files. Each thread has its own: stack, registers, PC, thread ID. Called Lightweight Process (LWP) — cheaper to create/switch than a full process.

Many-to-One (M:1)
  • Many user threads → 1 kernel thread
  • Fast creation, no kernel overhead
  • One block → all block; no true parallelism
  • Rarely used in modern systems
One-to-One (1:1)
  • Each user thread has its own kernel thread
  • True parallelism on multiple cores
  • One block doesn't affect others
  • Used by: Windows, Linux, macOS
  • Thread vs Process: Thread = lightweight, shares memory with siblings. Process = heavyweight, own memory space.
  • When prefer thread: tasks share data (web server worker threads), need responsiveness (UI + background work)
  • When prefer process: tasks need isolation (crash of one shouldn't affect others), different privileges
  • Concurrency vs Parallelism: Concurrency = multiple tasks making progress on 1 CPU (illusion). Parallelism = multiple tasks truly executing simultaneously on multiple CPUs.

Amdahl's Law

Speedup ≤ 1 / (S + (1-S)/N) S = serial fraction of program (cannot be parallelized) N = number of processor cores Example: 75% parallel (S=0.25), N=2 cores: 1/(0.25 + 0.75/2) = 1/0.625 = 1.6× As N→∞: speedup approaches 1/S (serial part always bottlenecks) Key insight: even 10% serial code limits max speedup to 10× regardless of core count
MEM

Memory Management

Address Types and MMU

Logical (Virtual) Address
  • Generated by CPU; what program sees
  • Range 0 to (process size − 1)
  • Translated to physical by MMU hardware
Physical Address
  • Actual RAM chip address
  • Programs never see this directly
  • MMU: physical = logical + base register (in contiguous allocation)

Fragmentation

Internal Fragmentation
  • Wasted space inside an allocated block
  • Process allocated more than it needs
  • Occurs in: fixed partitions, paging (last page may be partial)
  • Example: 100KB partition, process needs 90KB → 10KB wasted inside
External Fragmentation
  • Wasted space between allocated blocks
  • Total free memory is enough but not contiguous
  • Occurs in: variable partitions, segmentation
  • Solutions: compaction (defragmentation), or use paging
Allocation AlgoRuleResult
First FitFirst hole big enoughFast; may fragment start of memory
Best FitSmallest hole big enoughTiny leftover holes; slow (full scan)
Worst FitLargest holeLarge leftover holes; worst overall

Paging — Address Translation

Key formulas: Page Size = Frame Size = 2^(offset bits) bytes Offset bits = log₂(Page Size) Page number bits = log₂(Process Size) − offset bits Frame number bits = log₂(RAM Size) − offset bits Logical address = [page# | offset] (total bits = log₂(Process Size)) Physical address = [frame# | offset] (total bits = log₂(RAM Size)) Address translation: Given logical address LA, page_size P: page# p = LA / P (integer division) offset d = LA mod P Look up p in page table → get frame# f Physical address = f × P + d (or: f * 2^offset_bits + d)
  • Most common exam problem — paging address translation [in EVERY exam paper]
  • Step 1: find page size (if not given — derive from bits or use sqrt(2×S×e) formula)
  • Step 2: compute offset and page# bits from page size and process size
  • Step 3: convert logical address to page#:offset
  • Step 4: look up page table for frame#
  • Step 5: physical address = frame# × page_size + offset

TLB (Translation Look-Aside Buffer)

Without TLB: paging requires 2 memory accesses (page table + data). TLB is a small, fast hardware cache of recent page→frame translations. On TLB hit: 1 memory access total. On TLB miss: access page table in memory (2 accesses), update TLB.

Effective Access Time (EAT): EAT = h×(TLB_time + mem_time) + (1-h)×(TLB_time + 2×mem_time) where h = TLB hit rate Example: h=0.80, TLB=20ns, mem=100ns: EAT = 0.8×(120) + 0.2×(220) = 96 + 44 = 140ns
  • Why TLB is small: associative memory (expensive); spatial/temporal locality means small TLB still gets ~95% hit rate
  • TLB full → replace entry using LRU or FIFO. Each process may need its own TLB entries → flush on context switch (or use ASIDs)

Two-Level Page Table

Purpose: A large address space needs a huge page table. Instead, page the page table itself (hierarchical). Address format: | outer index p1 | inner index p2 | offset d | Steps: 1. offset bits = log₂(Page Size) 2. total page# bits = log₂(ProcessSize) − offset bits 3. inner index bits = log₂(PageSize / entry_size) [entries per page-table page] 4. outer index bits = total page# bits − inner index bits Example: Process=32MB, RAM=16MB, Page=2KB, entry=4B Offset = log₂(2KB) = 11 bits Page# bits = log₂(32MB) − 11 = 25 − 11 = 14 bits Inner page index = log₂(2KB/4B) = log₂(512) = 9 bits Outer page index = 14 − 9 = 5 bits Logical address: | 5 bits | 9 bits | 11 bits |
  • Two-level page table: appears in almost every exam. Know the 4-step derivation above cold.
  • Draw the 3-level table structure: outer PT → row of inner PTs → data frames
  • Max file size in inode: direct + single indirect + double indirect + triple indirect

Segmentation

Logical address = (segment#, offset). Segment table: each entry has base (physical start) and limit (segment length). If offset ≥ limit → protection fault. Segments are variable-size → external fragmentation, unlike paging.

Advantage over paging: matches programmer's view of memory (code segment, stack segment, data segment). Sharing at segment level (two processes can share a code segment).

VMEM

Virtual Memory and Page Replacement

Demand Paging

Load pages into RAM only when accessed. Most programs have locality of reference — they access a small subset of pages at any time. Valid-invalid bit in page table: valid=1 (page in RAM), valid=0 (page on disk → causes page fault).

Page Fault Handling Steps

CPU accesses page with valid-bit = 0 → hardware page-fault trap to OS
OS checks: valid address for this process? If not → abort. If yes but not in RAM → continue
Find a free frame. If none available → run page replacement algorithm to pick victim frame
If victim frame is dirty (modified) → write to disk (swap out). Update victim's page table entry to invalid.
Load required page from disk into free frame. Update page table: set frame# and valid bit = 1
Restart the instruction that caused the fault

Thrashing: process spends more time paging than executing. Cause: too few frames → constant page faults. Effect: low CPU utilization + high I/O → OS adds more processes → makes it worse. Solution: working set model, adjust degree of multiprogramming.

Page Replacement Algorithms

AlgorithmReplaceFaults (3 frames, 7,0,1,2,0,3,0,4,2,3...)Notes
FIFOPage in memory longest (oldest)15Simple; Belady's Anomaly: more frames can → more faults!
Optimal (OPT)Page not needed for longest time in future9Best possible; theoretical only (needs future knowledge)
LRUPage not used for longest time in past12No Belady's Anomaly; generally good; expensive to implement exactly
Clock (2nd Chance)FIFO, but give pages with reference bit=1 a second chance (clear bit, skip)≈LRUPractical LRU approximation
  • Page replacement table is in every exam — draw frame states row by row
  • Mark each access: H (hit, already in frames) or F (fault, must load)
  • On fault: if frames not full → fill. If full → apply replacement rule
  • LRU: on fault, replace the page whose LAST USE was furthest back in time (scan left in history)
  • Optimal: on fault, look FORWARD — replace the page that appears latest in the future reference string (or not at all)
  • Enhanced Clock: ordered pairs (reference, modify): (0,0) replace first, then (0,1), then (1,0), then (1,1)
  • Global replacement: process can steal frames from other processes (higher throughput but variable performance)
  • Local replacement: process replaces only from its own allocated frames (consistent performance)
DISK

Disk Scheduling

Disk Structure

Seek TimeTime to move disk arm to target cylinder. Dominates total access time.
Rotational LatencyTime waiting for target sector to rotate under head. Avg = half rotation time.
Transfer TimeTime to read/write the actual sector data.
Cylinder SkewOffset between sector-0 positions on adjacent cylinders. Compensates for seek time.
MBR (Sector 0)Master Boot Record. Contains boot code + partition table (start, size of each partition).
Bad SectorHandled by sector sparing (map to spare) or sector forwarding (shift sectors). Detected by ECC during format.

Disk Scheduling Algorithms

AlgorithmRuleMovement (queue: 98,183,37,122,14,124,65,67, head=53)Notes
FCFSService in arrival order640 cylindersFair; very inefficient head movement
SSTFService nearest cylinder to current head236 cylindersMuch better; can starve far cylinders
SCAN (Elevator)Move in one direction, service all, reverse at end208 cylindersNo starvation; travels to disk end even if no requests there

C-LOOKLike C-SCAN but only goes to last request (not disk end), then jumps to lowest request—Best practical; eliminates unnecessary travel to disk ends
  • Disk scheduling calculation appears in every exam. Must show order of serviced cylinders AND total head movement.
  • SSTF: at each step, find the cylinder CLOSEST to current head position. Ties: break by direction or problem convention.
  • SCAN: go in declared direction, service all requests, reverse when no more requests in that direction (LOOK) or at disk end (SCAN)
  • C-LOOK from head at 53, moving up (queue: 65,67,98,122,124,183 | lower: 37,14): Service upward: 65,67,98,122,124,183 → jump to 14 → service upward: 14,37. Movement = (183−53) + (183−14) = 130 + 169 = 299
  • Recommended default: SSTF (common) or C-LOOK (best practical)
FS

File Systems

File Allocation Table (FAT)

An array where FAT[i] tells you the NEXT block in the chain for the file using block i. Values: 0 = free block; next_block_number = next in chain; EOF marker = last block of file. Directory entry stores only the first block number; full chain is reconstructed via FAT.

Example: File starts at block 4, size = 3 blocks. FAT: [ 0, 0, 0, 0, 7, 0, 0, 9, 0, EOF, 0 ... ] ↑ ↑ ↑ ↑ Chain: 4 → FAT[4]=7 → FAT[7]=9 → FAT[9]=EOF Reading: read block 4, read block 7, read block 9, done. To find n-th block: follow chain n-1 times (sequential access = linear time) Random access: must traverse chain from start to reach block n (slower than indexed)
  • Draw FAT diagram for given file sizes and block sizes
  • Number of blocks needed = ceil(file_size / block_size)
  • How to find the 10th block of a file in FAT? Start at first block, follow 9 pointers in FAT chain.
  • Linked allocation (FAT) random access problem: must traverse from start. Indexed allocation (inode) solves this.

Unix inode Structure

Each file has one inode storing: permissions, owner, size, timestamps, and block pointers. The filename lives in the directory, NOT in the inode.

inode block pointer structure: 12 direct pointers → each points to a data block 1 single indirect → points to a block of pointers 1 double indirect → pointer → block of pointers → data blocks 1 triple indirect → 3 levels of pointer blocks Maximum file size: Let B = block size (bytes), P = pointer size (bytes) Pointers per block = B/P Direct: 12 × B Single indirect: (B/P)¹ × B Double indirect: (B/P)² × B Triple indirect: (B/P)³ × B Total = B × [12 + (B/P) + (B/P)² + (B/P)³] Example: B = 8KB = 8192 bytes, P = 2 bytes → B/P = 4096 Direct: 12 × 8KB = 96 KB Single: 4096 × 8KB = 32 MB Double: 4096² × 8KB = 128 GB Triple: 4096³ × 8KB = 512 TB Max ≈ 512 TB
  • inode max file size calculation — in EVERY exam since 2018
  • Most common setup: 4KB blocks, 4-byte pointers (1024 pointers/block) OR 8KB blocks, 2-byte pointers (4096 pointers/block)
  • File /etc/config/host.txt traversal: root inode → root dir block → find "etc" entry → etc inode → etc dir block → find "config" → config inode → config dir block → find "host.txt" → host.txt inode → data blocks

Hard Link vs Soft Link

Hard Link (ln f1 f2)
  • Another directory entry pointing to same inode
  • File deleted only when ALL hard links removed (reference count = 0)
  • Cannot cross file systems or link to directories
  • Deleting original: other hard links still work
Soft Link (ln -s f1 f2)
  • Special file containing a path to another file
  • If original deleted: soft link becomes dangling (broken)
  • Can cross file systems; can link to directories
  • Flexible but fragile
