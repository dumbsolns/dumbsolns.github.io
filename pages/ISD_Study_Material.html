Q1 · ISD

Information Gathering — Interviews

Every exam · Same 5 sub-questions
The Q1 scenario changes (hospital system, library system, MIS for automotive company) but the five sub-questions are IDENTICAL every time. Learn the template, adapt to the scenario.

Sub-question (a) — Five Interview Goals [5 marks]

Write 5 goals that examine different dimensions of the current system and the desired new system. Always cover these 5 angles:

1. Current StateUnderstand how the current system works, its processes, data flows, and who uses it. "Understand the current patient record management workflow."
2. Desired FeaturesIdentify what new functionality stakeholders want. "Determine desired features such as automated appointment scheduling."
3. Challenges / Pain PointsIdentify problems and frustrations with the current system. "Identify challenges faced by doctors in accessing patient history quickly."
4. Benefits / Success CriteriaDefine what success looks like. "Assess the potential benefits including time savings and error reduction."
5. Integration / ConstraintsUnderstand technical constraints and integration needs. "Evaluate requirements for integrating with existing medical record systems."
Write goals as complete statements starting with a verb: "Understand...", "Identify...", "Determine...", "Assess...", "Evaluate...". Each goal should be one sentence tied specifically to the scenario given.

Sub-question (b) — Stakeholder Interview Strategy [5 marks]

Explain WHO you will interview and WHY (justification). Always use a top-down approach — start with management, then operations, then end users.

Top ManagementE.g., Chief Medical Officer, Hospital Director. Goal: strategic objectives, budget constraints, high-level requirements, organizational goals.
Department HeadsE.g., Head of IT, Department Heads. Goal: operational processes, departmental requirements, existing workflows.
Direct Users (Operational)E.g., Doctors, Nurses. Goal: daily operational challenges, specific functional needs, usability requirements.
End Users / ClientsE.g., Patients, Students. Goal: user experience, pain points from the client perspective, desired features.

Justify each choice: "The Chief Medical Officer is interviewed first because they define strategic priorities and constraints that must shape all subsequent decisions."

Sub-question (c) — Interview Structure [5 marks]

Every exam specifies using a semi-structured approach. Always answer this way:

Chosen structure: SEMI-STRUCTURED interview Rationale: • Combines prepared questions (structure) with flexibility to probe deeper (unstructured) • Prepared questions ensure all required topics are covered systematically • Open-ended follow-up questions allow exploration of unexpected issues • Maintains focus while adapting to the interviewee's knowledge and experience • Reduces bias: no leading questions, but interviewer can clarify ambiguities Structure of the interview: 1. Opening: introduce purpose, build rapport, explain recording method 2. Core prepared questions: cover all 5 interview goals (a mix of open and closed) 3. Probing follow-ups: "Can you give me an example?" / "Why is that?" 4. Closing: summarize key points, ask for additions, schedule follow-up Alternative structures (know these for comparison): • Structured: all questions fixed in advance. Good for quantitative data, surveys. • Unstructured: free-flowing conversation. Good for initial exploration only. → Semi-structured is preferred because it balances rigor with flexibility.

Sub-question (d) — Five Open-Ended Questions [5 marks]

Write 5 questions you would email before the interview to let the interviewee prepare. All must be open-ended (no yes/no answers possible). Adapt to scenario:

Template questions (adapt nouns to the scenario): 1. "What are the most significant challenges you currently face in managing [patient records / library resources / information systems]?" 2. "What features or functionalities would you like to see in the new [hospital management / library / MIS] system?" 3. "How does the current system impact your ability to make timely and informed decisions?" 4. "What aspects of the current system work well and should be preserved in the new system?" 5. "How do you envision the new system improving collaboration and communication among [doctors and nurses / staff and students]?"
  • All 5 must be open-ended — starting with "What", "How", "Describe", "Explain", "Why"
  • Each question should target a different interview goal from sub-question (a)
  • Questions must be relevant to the specific scenario given in the exam

Sub-question (e) — Interviewing Subordinates/Colleagues [5 marks]

Answer: Yes, you would interview subordinates. Then explain how.

YES — interviewing subordinates and colleagues is important because: • They are daily users of the system — know operational realities • Management may not be aware of ground-level problems • Broader range of perspectives → more complete requirements • Increases buy-in: users feel heard → better adoption How to conduct: • Conduct FOCUS GROUPS: gather 5–8 similar users in one session (e.g., all nurses together, all patients together) → Encourages discussion, reveals shared problems → More time-efficient than individual interviews • Alternatively use SURVEYS/QUESTIONNAIRES for large groups → Reach many users efficiently → Good for quantitative data (how often? how many?) • Individual interviews for key users with specialized roles • Ensure anonymity where concerns about management retaliation exist Approach: semi-structured, same interview goals as senior interviews, but focused on day-to-day operational realities rather than strategy
Q2 · CPM

Critical Path Method (CPM) & PERT vs Gantt

Fall 2024 Q2

CPM — Critical Path Calculation

Given: Task list with effort (person-days) and prerequisites Step 1: Build precedence network Draw nodes (tasks) and directed arrows (dependencies) Start node and End node connect everything Step 2: Forward Pass — compute Earliest Start (ES) and Earliest Finish (EF) ES of Task A (no prerequisites) = 0 EF = ES + Duration ES of any task = max(EF of ALL predecessors) Step 3: Backward Pass — compute Latest Finish (LF) and Latest Start (LS) LF of last task = EF of last task (project deadline) LS = LF - Duration LF of any task = min(LS of ALL successors) Step 4: Compute Slack (Float) for each task Slack = LS - ES (or equivalently LF - EF) Step 5: Identify Critical Path Critical tasks: Slack = 0 Critical Path = the longest path from Start to End through critical tasks

Worked Example (Fall 2024 Q2)

Tasks: A: Requirements Analysis (10 days, no prereqs) B: Database Design ( 8 days, prereq: A) C: User Interface Design ( 7 days, prereq: A) D: Backend Development (15 days, prereq: B) E: Frontend Development (12 days, prereq: C) F: System Integration (10 days, prereqs: D, E) G: User Acceptance Testing ( 5 days, prereq: F) H: Deployment ( 3 days, prereq: G) Forward Pass: A: ES=0, EF=10 B: ES=10, EF=18 (after A) C: ES=10, EF=17 (after A) D: ES=18, EF=33 (after B) E: ES=17, EF=29 (after C) F: ES=33, EF=43 (after D AND E; max(33,29)=33) G: ES=43, EF=48 H: ES=48, EF=51 ← Project Duration = 51 days Backward Pass (LF of H = 51): H: LF=51, LS=48 G: LF=48, LS=43 F: LF=43, LS=33 D: LF=33, LS=18 → Slack = 18-18 = 0 ✓ CRITICAL E: LF=33, LS=21 → Slack = 21-17 = 4 (not critical) B: LF=18, LS=10 → Slack = 0 ✓ CRITICAL C: LF=21, LS=14 → Slack = 14-10 = 4 (not critical) A: LF=10, LS=0 → Slack = 0 ✓ CRITICAL Critical Path: A → B → D → F → G → H (total = 10+8+15+10+5+3 = 51 days) Impact of Task E delayed by 4 days: E now finishes at day 29+4=33 (same as D!) F still starts at max(33,33)=33 → NO IMPACT on project completion (E's slack = 4 days; a 4-day delay exactly uses up slack, no extension) If E delayed by 5 days → project extends by 1 day (51→52)
  • Draw the network diagram — nodes for each task, arrows for dependencies, ES/EF/LS/LF inside each node
  • If a task is delayed by X days: if X ≤ slack → no impact. If X > slack → project extends by (X - slack) days
  • PERT vs Gantt [Fall 2024 Q2c]: PERT better for initial scoping (shows dependencies, critical path); Gantt better for tracking progress (visual, easy to update and read for management)

PERT vs Gantt Chart Comparison

PERT Diagram
  • Network of nodes and arrows
  • Shows task dependencies and precedence
  • Identifies critical path and slack time
  • Better for complex, interconnected projects
  • Better for initial project scoping
  • Harder for non-technical stakeholders to read
Gantt Chart
  • Horizontal bar chart against timeline
  • Shows task duration and schedule
  • Easy to see current progress vs plan
  • Does NOT show dependencies
  • Better for tracking progress
  • Easy for all stakeholders to understand
Q2 · Finance

Cost-Benefit Analysis — NPV, ROI, Payback, Ranking

Spring 2023 & Spring 2024 Q2

Four Financial Analysis Metrics

For each project, given: Year 0 to Year N cash flows (+income, -expenditure) and a discount rate i% 1. NET PROFIT = Total Income - Total Expenditure (sum all years, no discounting) 2. PAYBACK PERIOD = Year when cumulative cash flow first becomes positive (Cumulative = running sum of all cash flows up to that year) 3. ROI (Return on Investment) = Net Profit / Total Investment × 100% Total Investment = sum of all negative cash flows (Year 0 expenditures) 4. NPV (Net Present Value): NPV = Σ [CFₙ / (1 + i)ⁿ] for n = 0 to N Where: CFₙ = cash flow in year n i = discount rate (as decimal: 8% = 0.08) n = year number Year 0 cash flow is NOT discounted (already at present value) CF₁ discounted: CF₁ / (1.08)¹ CF₂ discounted: CF₂ / (1.08)² If NPV > 0: project is financially worthwhile (returns more than cost of capital) If NPV < 0: project destroys value (don't proceed)

Worked Example (Spring 2023 — Project 1: i=8%)

Year 0: -90,000 Year 1: +10,000 Year 2: +10,000 Year 3: +10,000 Year 4: +20,000 Year 5: +100,000 Net Profit = (10+10+10+20+100) - 90 = 150,000 - 90,000 = $60,000 Cumulative Cash Flow: End of Year 0: -90,000 End of Year 1: -90,000 + 10,000 = -80,000 End of Year 2: -80,000 + 10,000 = -70,000 End of Year 3: -70,000 + 10,000 = -60,000 End of Year 4: -60,000 + 20,000 = -40,000 End of Year 5: -40,000 + 100,000 = +60,000 ← PAYBACK at Year 5 Payback Period = 5 years (or more precisely: 4 + 40,000/100,000 = 4.4 years) ROI = 60,000 / 90,000 × 100% = 66.7% NPV at 8%: Year 0: -90,000 / 1.000 = -90,000.00 Year 1: 10,000 / 1.080 = 9,259.26 Year 2: 10,000 / 1.166 = 8,573.39 Year 3: 10,000 / 1.260 = 7,938.32 Year 4: 20,000 / 1.360 = 14,701.48 Year 5: 100,000 / 1.469 = 68,058.32 NPV = -90,000 + 9,259 + 8,573 + 7,938 + 14,701 + 68,058 = $18,530

Ranking Projects & Which Analysis Suits Which Situation

Ranking order (generally): Rank by NPV first (accounts for time value of money — most rigorous) If tied or close: use ROI as tiebreaker Which analysis suits which situation: NPV: Best for comparing projects with different timelines and scales. Accounts for time value of money. Most theoretically correct. Use when: long projects, significant capital investment. Payback Period: Best when liquidity is critical (need money back fast). Simpler, easy to understand. Ignores cash flows after payback. Use when: company has cash flow constraints, risky environment. ROI: Best for comparing efficiency of investment across different sizes. Easy to compare as a percentage. Ignores time value. Use when: quick comparison of projects of different scales. Net Profit: Simplest — ignores time value. Good for initial screening only. Use when: quick first pass to eliminate clearly unprofitable projects.
  • Always show the full cumulative cash flow table for Payback Period — don't just state the year
  • Show NPV calculation year by year with the discount factor for each year
  • For ranking: state your criteria clearly. Usually: NPV > 0 → viable; highest NPV = best; then ROI to differentiate
  • Spring 2024 Q2b: "Explain which analysis suits which situation" — write 2–3 sentences for each of the 4 metrics
Q2 · DFD

DFDs, Decision Trees & Decision Tables

Fall 2022 Q2

Drawing DFDs from Requirements (Context → Level 2)

DFD Notation: External Entity: Rectangle (source or sink of data, outside system boundary) Process: Circle/Bubble (transforms input data into output data) Data Store: Two parallel horizontal lines (data at rest — files, databases) Data Flow: Named directed arrow (movement of data between elements) DFD Rules (must follow): ✗ Data flow CANNOT go directly between two external entities ✗ Data flow CANNOT go directly between two data stores ✗ External entities CANNOT connect directly to data stores ✓ All processes must have at least one input AND one output ✓ All data flows must be named Level hierarchy: Context (Level 0): entire system = 1 bubble, all external entities, no data stores Level 1: explode to 3–7 major subsystem processes + data stores Level 2: explode each Level 1 process into sub-processes
For the travel booking exam question: External entities = Traveler, Airlines, Hotels, Payment System. Major processes = Booking Management, Customer Management, Payment Processing, Notification. Data stores = Customer DB, Booking DB, Payment Records. Context: Traveler ←→ [Travel Booking System] ←→ Airlines, Hotels, Payment System

Decision Trees

A decision tree is a graphical representation of decision logic. Start from the root (first decision), branch on each condition, end at leaf nodes (outcomes/actions).

Travel Booking discount example: Condition 1: Is customer a Senior Citizen (age ≥ 60)? YES → 15% discount on ALL booking types → END NO → Condition 2: Is customer a Student (age 18–25)? YES → Condition: Is it a vacation package? YES → 10% discount NO → Condition: Group booking (≥5 people)? YES → 20% discount NO → 5% standard discount NO → Condition: Group booking (≥5 people)? YES → 20% discount NO → 5% standard discount

Decision Tables

A decision table lists all combinations of conditions and the corresponding actions. More systematic than a decision tree for many conditions.

Structure: ┌─────────────────┬───┬───┬───┬───┬───┬───┬───┬───┐ │ Conditions │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │ 8 │ ├─────────────────┼───┼───┼───┼───┼───┼───┼───┼───┤ │ Senior Citizen │ Y │ N │ N │ N │ N │ N │ N │ N │ │ Student (18-25) │ - │ Y │ Y │ Y │ N │ N │ N │ N │ │ Vacation Package│ - │ Y │ N │ N │ - │ - │ - │ - │ │ Group (≥5) │ - │ - │ Y │ N │ Y │ N │ - │ - │ ├─────────────────┼───┼───┼───┼───┼───┼───┼───┼───┤ │ Actions: │ │ │ │ │ │ │ │ │ │ 15% discount │ X │ │ │ │ │ │ │ │ │ 10% discount │ │ X │ │ │ │ │ │ │ │ 20% discount │ │ │ X │ │ X │ │ │ │ │ 5% discount │ │ │ │ X │ │ X │ │ │ └─────────────────┴───┴───┴───┴───┴───┴───┴───┴───┘ (Y=Yes, N=No, -=Don't care, X=apply this action)
Q3 · SE

Software Engineering Concepts

Every exam · Answer any 4–5 from ~6 options

Refactoring (XP) [Fall 2024, Fall 2022, Spring 2024]

Refactoring is the process of restructuring existing code to improve its internal structure (readability, maintainability, reduced complexity) without changing its external behavior.

Why it is NOT a waste of time: • Prevents code decay: each change makes code cleaner, not messier • Reduces technical debt: small refactors now prevent large rewrites later • Makes adding new features easier and faster • Improves team understanding of the codebase • Removes duplicated code — one change in one place instead of many Simple code example: BEFORE refactoring (hard to maintain): if (age > 60) { discount = 0.15; } if (age >= 18 && age <= 25) { discount = 0.10; } if (groupSize >= 5) { discount = 0.20; } AFTER refactoring (readable, maintainable): function calculateDiscount(customer) { if (customer.isSenior()) return SENIOR_DISCOUNT; if (customer.isStudent()) return STUDENT_DISCOUNT; if (customer.isGroupBooking()) return GROUP_DISCOUNT; return STANDARD_DISCOUNT; }

Verification vs Validation [Fall 2024, Spring 2023, Fall 2022]

Verification — "Are we building the product RIGHT?"
  • Checks that software conforms to its specification
  • Internal process — developer checks against design documents
  • Activities: code reviews, walkthroughs, inspections, unit testing
  • Asks: does the code do what the design says it should?
  • Can be done without running the program
Validation — "Are we building the RIGHT product?"
  • Checks that software meets the actual needs of the customer
  • External process — customer/user evaluates the system
  • Activities: acceptance testing, user testing, beta testing
  • Asks: does the system do what the customer actually needs?
  • Requires running the system with real users
  • Who performs validation? [Spring 2023 Q3c] — The SOFTWARE USER (customer/client), not the developer. Developer performs verification. Validation test = acceptance test conducted by or with the user.

Spiral vs V-Model [Fall 2024 Q3a]

Spiral Model
  • Iterative with 4 quadrants per cycle: plan → risk analysis → develop → review
  • Explicit risk analysis every cycle — defines it
  • Multiple iterations — progressively detailed product
  • Best for: large, complex, high-risk projects
  • Testing is incremental throughout all cycles
  • Changes easily accommodated between spirals
  • Suitable for safety-critical systems (medical software)
V-Model
  • Extension of Waterfall — sequential phases
  • No explicit risk management
  • Each dev phase paired with a test phase (mirror image)
  • Best for: well-defined, stable requirements
  • Testing planned early but executed late
  • Poor adaptability to requirement changes
  • Suitable for medical device firmware (stable spec + rigorous testing)

Testing vs Debugging [Fall 2024 Q3f, Fall 2022]

Testing
  • Executes the software to FIND errors
  • Performed throughout development
  • Uses test cases (white box / black box)
  • Objective: discover that errors EXIST
  • Can be done by developers or separate testers
  • Planned activity with documented test cases
Debugging
  • Locates, diagnoses, and FIXES a known error
  • Triggered by a failed test
  • Uses debuggers, logging, code inspection
  • Objective: find WHERE the error is and fix it
  • Usually done by the developer who wrote the code
  • Unplanned — happens in response to test failures

White Box vs Black Box Testing [Spring 2023 Q3e, Fall 2022]

WHITE BOX (Structural / Glass Box): Knowledge: Full access to source code Tests: Designed to execute every statement, branch, and path Who: Developer Tools: Code coverage analyzers, cyclomatic complexity Cyclomatic Complexity V(G) = E - N + 2P (edges - nodes + 2) = number of predicate nodes + 1 = number of enclosed regions + 1 → Gives minimum number of test cases for full path coverage BLACK BOX (Functional / Specification-based): Knowledge: No access to code — only input/output specification Tests: Designed from requirements spec Who: Tester or end user Techniques: Equivalence partitioning, boundary value analysis Equivalence Partitioning: Valid input: 1–100 → test with 50 (representative of valid class) Invalid: <1 → test with 0 or -1; >100 → test with 101 Boundary Value Analysis: Test at: 0, 1, 2 (near lower bound) and 99, 100, 101 (near upper bound)

XP Pair Programming [Fall 2022 Q3b]

Pair programming is an XP practice where two developers share a single workstation. One is the driver (types code), the other is the navigator (reviews in real time, catches errors, thinks strategically). They switch roles frequently.

  • Code is reviewed continuously — fewer defects reach testing
  • Knowledge sharing — both understand the codebase equally
  • Collective ownership — both own the code written together
  • Counterintuitive: two people at one machine is NOT half as productive — studies show ~15% slower but 15–50% fewer defects, saving more time in testing and maintenance

Framework Activities + Umbrella Activities [Fall 2022 Q3e, Q3f]

5 Framework Activities (apply to every software process model): 1. Communication — understand stakeholder requirements 2. Planning — estimate, schedule, identify risks 3. Modeling — analysis models + design models 4. Construction — code + unit/integration testing 5. Deployment — deliver, evaluate, feedback 8 Umbrella Activities (run across all framework activities): 1. Software project tracking and control 2. Risk management 3. Software quality assurance (SQA) 4. Technical reviews (formal) 5. Software configuration management 6. Work product preparation and production 7. Reusability management 8. Measurement (metrics)

Sprint Planning Meeting (Scrum) [Fall 2024 Q3e]

Sprint Planning Meeting — what it is: Meeting at the START of each sprint (1–2 day meeting for 2-4 week sprint) Attendees: Product Owner, Scrum Master, Development Team INPUTS to Sprint Planning: 1. Product Backlog (prioritized list of all features/user stories) 2. Team Velocity (how many story points team completed in previous sprint) 3. Team capacity (who is available, how many days/hours this sprint) 4. Definition of Done (agreed criteria for a story to be "complete") KEY OUTPUTS from Sprint Planning: 1. Sprint Goal (a short statement of what the sprint aims to achieve) 2. Sprint Backlog (the specific user stories selected for this sprint) 3. Task breakdown (user stories broken into individual tasks with estimates) 4. Commitment (team commits to delivering the Sprint Backlog)

Waterfall Mid-Project Change [Spring 2023 Q3a]

Scenario: Client using Waterfall insists on significant requirement changes midway Response strategy: 1. Assess impact: document what must change in design, code, tests, docs 2. Change Control Process: formal change request with cost/time impact analysis 3. Negotiate scope: discuss what can be included vs. deferred to Phase 2 4. Update project plan: revise schedule, resources, milestones 5. Consider switching model: if changes are frequent → suggest Agile (Scrum) which accommodates change by design (sprint-by-sprint priorities) 6. Baseline and document: all parties sign off on revised requirements Why Waterfall struggles with changes: • Later phase changes are exponentially more expensive (defect cost curve) • Previous deliverables may need rework (cascading changes) • Contract and schedule commitments may be violated

Three Projects Suitable for Spiral [Spring 2024 Q3d]

  • Medical Software (e.g., patient management system): High risk to human safety; requirements evolve as medical protocols change; needs formal risk analysis every iteration
  • Defense/Military Systems: Extremely high cost of failure; classified requirements may change; risk analysis critical; budget is large
  • Financial Trading Platform: High business risk; regulatory requirements change; performance requirements uncertain until tested; multiple stakeholders with competing needs

Continuous Integration (XP) [Spring 2024 Q3c]

Continuous Integration (CI) is the XP practice of integrating all developers' code into a shared mainline multiple times per day, with an automated build and test suite running after every integration.

  • Problems are found immediately — when code is fresh in developer's mind
  • No "integration hell" at project end
  • Working software is always available
  • Build broken → must be fixed before any new work proceeds

Agile (Scrum) vs Waterfall Comparison [Spring 2024 Q3a]

Agile / Scrum
  • Iterative sprints (2–4 weeks)
  • Working software every sprint
  • Requirements change welcome
  • Customer continuously involved
  • Self-organizing cross-functional team
  • Best: unclear/changing requirements
Waterfall
  • Sequential phases, no going back
  • Working software only at end
  • Change = expensive and disruptive
  • Customer only at start and end
  • Specialized roles per phase
  • Best: clear, stable requirements
Q4 · UML

UML Diagrams — Use Case, Sequence, Class, Activity

Every exam · Answer any 3 from 4 options
The scenario changes every exam (hospital patient management, student management, car rental, supermarket website, university course registration) but the SAME four diagram types are asked every time. Learn the notation for each diagram type cold, then apply to any scenario.

(a) Use Case Diagram

Components: Actor (stick figure): A user or external system interacting with the system Use Case (oval): A specific function the system provides System boundary (rect): Box enclosing all use cases Association (line): Actor participates in a use case «include» (dashed →): Use case A ALWAYS includes behavior of B (mandatory) «extend» (dashed →): Use case A OPTIONALLY extends B (conditional) Generalization (△ line): Actor or use case inherits from another Hospital Patient Management example: Actors: Patient, Doctor, Nurse, Admin Staff, Medical Records System Use Cases: Register Patient, Book Appointment, View Medical History, Prescribe Treatment, Schedule Nursing Care, Generate Bill, Update Medical Records, Send Notifications Relationships: Patient ──── Register Patient Patient ──── Book Appointment Doctor ──── View Medical History Doctor ──── Prescribe Treatment Prescribe Treatment «include» Update Medical Records (always happens) Book Appointment «extend» Send Notification (happens when booking confirmed)

(b) Sequence Diagram

Components: Objects (boxes at top): Name:Class or just Name Lifeline (dashed line): Vertical line below each object showing time Activation bar (thin rect): Object is active/executing Message (solid arrow →): Method call or synchronous message Return (dashed arrow ←): Return value from a method call Time flows DOWNWARD Patient booking appointment example: Objects: Patient, System, Doctor, Nurse Sequence: Patient → System: requestAppointment(patientId, preferredDate) System → Doctor: checkAvailability(preferredDate) Doctor →→ System: return availableSlots[] System → Patient: displayAvailableSlots(slots) Patient → System: selectSlot(slotId) System → Doctor: sendBookingRequest(patientId, slotId) Doctor →→ System: confirmBooking() System → Nurse: scheduleVisit(patientId, confirmedSlot) Nurse →→ System: acknowledgement() System → Patient: sendConfirmation(bookingDetails)

(c) Class Diagram

Each class: 3-compartment box ┌──────────────────┐ │ ClassName │ ← class name (bold, centered) ├──────────────────┤ │ -attributeName:Type │ ← attributes (- private, + public, # protected) ├──────────────────┤ │ +methodName():Return│ ← methods └──────────────────┘ Relationships: ───── Association "A uses B" (plain line, with label + multiplicity) ────◇ Aggregation "A has B" (hollow diamond at A) ────◆ Composition "A contains B" (solid diamond at A; B dies with A) ────▷ Generalization "A is-a B" (hollow triangle at B, superclass end) - - → Dependency "A depends on B" (dashed arrow) Hospital example: Patient ────── Appointment (1 patient has 0..* appointments) Appointment ────── Doctor (1 appointment has exactly 1 doctor) Appointment ────── MedicalRecord (1 appointment may generate 1 record) Doctor ──▷ StaffMember (Doctor IS-A StaffMember) Nurse ──▷ StaffMember (Nurse IS-A StaffMember) Patient class: +patientId: String +name: String -dateOfBirth: Date +bloodGroup: String +getAge(): int +bookAppointment(d: Date): Appointment

(d) Activity Diagram

Components: ● Start node (filled black circle) (activity) Rounded rectangle — one activity/action ◇ Decision node (diamond) — conditional branch [condition] ═══ Fork bar — splits into PARALLEL branches (synchronization bar) ═══ Join bar — merges parallel branches back ⊙ End node (circle with inner dot) Patient checking in for treatment example: ● START ↓ [Arrive at Hospital] ↓ [Register at Reception] ↓ ◇ Is patient registered? ──NO──→ [Create New Patient Record] ──┐ │YES ↓ └──────────────────────────────────────────────────────[Update Existing Record] ↓ [Wait in Waiting Area] ↓ ◇ Doctor available? ──NO──→ [Wait] ──→ (loop back) │YES ↓ ═══════ FORK ═════════════ ↓ ↓ [Doctor Reviews Records] [Nurse Takes Vitals] ↓ ↓ ════════ JOIN ═════════════ ↓ [Receive Treatment] ↓ [Generate Bill] ↓ [Patient Discharged] ↓ ⊙ END
  • Use case diagram is ALWAYS about system SCOPE — what the system does for each actor. Don't put implementation details in use cases.
  • Sequence diagram — always show return messages (dashed arrows back). Time goes DOWN.
  • Class diagram — always include multiplicity (1, 0..*, 1..*) on both ends of associations
  • Activity diagram — use fork/join bars whenever the exam mentions parallel/simultaneous processes
Q5 · SE Est.

Software Estimation — Functional Decomposition + LOC

Every exam · Same 3 sub-questions every time
Q5 asks the SAME three questions every exam with a different system. The scenario changes (DMS, food delivery, ride-sharing, social media platform, mobile app) but the method is identical. Master this template.

Sub-question (a) — Functional Decomposition [5 marks]

Decompose the software into its major functions, then break each into sub-functions. This produces the table used in part (b).

Template: Cloud Document Management System (DMS) — Fall 2024 Q5 Function Sub-functions ─────────────────────────────────────────────────────── 1. User Authentication Login, Registration, Password Reset 2. Document Upload/Storage Upload, Version Control, Storage Mgmt 3. Document Retrieval/Search Search, Filter, Preview 4. Document Sharing Share Links, Permission Control, Notifications 5. Collaboration Tools Comments, Co-editing, Activity Log 6. Admin/Management Dashboard User Management, Usage Analytics, Reports → These 6 major functions become the rows in the LOC estimation table (part b)

Sub-question (b) — Estimate Function Sizes in LOC [5 marks]

For each function identified in (a), use the 3-point estimation formula and provide LOC estimates.

Sub-question (c) — Calculate Effort and Cost [5 marks]

Given (from exam): Productivity = 700 LOC/person-month (Fall 2024) Labor rate = $8,000/person-month (Fall 2024) Step 1: Effort Effort = Total LOC / Productivity = 5,633 / 700 = 8.05 person-months ≈ 8 person-months Step 2: Cost Cost = Effort × Labor Rate = 8 × $8,000 = $64,000 Step 3: Duration (if team size given, e.g., 4 developers) Duration = Effort / Team_Size = 8 / 4 = 2 months Other exam examples: Spring 2023: 450 LOC/pm, $7,000/pm Spring 2024: 600 LOC/pm, $9,000/pm Fall 2022: 450 LOC/pm, $7,000/pm Always show all steps: Effort = LOC/Productivity → Cost = Effort × Rate
